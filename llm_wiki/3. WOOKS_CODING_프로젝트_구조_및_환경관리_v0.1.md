| 컴포넌트 | 폴더 | 역할 | 런타임 | 외부 노출 |

## 3. 환경변수(.env) 통합 관리

### 3.1 원칙
- **루트에 `.env` 단 하나**만 둔다. 컴포넌트별로 `.env`를 복제하지 않는다.
- 모든 컴포넌트(Django·FastAPI·Worker)는 **루트의 동일한 `.env`를 로드**한다.
- 커밋 대상은 `.env.example`(값 비움)만. 실제 `.env`는 `.gitignore`로 제외.
- 로컬 최초 설정: `cp .env.example .env` 후 값 채우기.

### 3.2 컴포넌트별 로딩 경로

각 컴포넌트는 자기 위치에서 루트 `.env`를 찾아 로드한다. `python-dotenv` 사용.

| 컴포넌트 | 로드 코드 위치 | 루트까지 상대 경로 |
|---|---|---|
| Django | `django/config/settings.py` | `BASE_DIR.parent / ".env"` |
| FastAPI | `fastapi_app/main.py` | `Path(__file__).resolve().parents[1] / ".env"` |
| Worker | `worker/main.py` | `Path(__file__).resolve().parents[1] / ".env"` |

**Django** (`config/settings.py`) — 현재 반영 완료:
```python
BASE_DIR = Path(__file__).resolve().parent.parent   # → django/
load_dotenv(BASE_DIR.parent / ".env")               # → 루트/.env
```

**FastAPI / Worker** (신규 작성 시 표준 패턴):
```python
from pathlib import Path
from dotenv import load_dotenv

ROOT = Path(__file__).resolve().parents[1]   # fastapi_app/ · worker/ → 루트
load_dotenv(ROOT / ".env")
```

> Docker 환경에서는 compose의 `env_file: .env` 또는 `environment:`로 주입하므로,
> `load_dotenv`는 파일이 없으면 조용히 무시된다(로컬/컨테이너 양쪽 호환).

### 3.3 `.env` 키 구성 (현재)
```
# Django
DJANGO_SECRET_KEY=...
DJANGO_DEBUG=True
DJANGO_ALLOWED_HOSTS=localhost,127.0.0.1

# PostgreSQL (공통)
POSTGRES_DB= / POSTGRES_USER= / POSTGRES_PASSWORD= / POSTGRES_HOST= / POSTGRES_PORT=

# (이후 추가) FastAPI/AI
# OPENAI_API_KEY=  INTERNAL_API_KEY=  CHROMA_HOST=  RAG_SCORE_THRESHOLD=0.35  TOP_K=5  CODE_TIMEOUT_SEC=5
```

> ⚠️ **주의 (빈 값 gotcha)**: `.env`에 `POSTGRES_DB=`처럼 키만 있고 값이 비면
> `os.environ.get("POSTGRES_DB", "기본값")`은 기본값이 아니라 **빈 문자열("")**을 반환한다.
> 로컬 실행 전 `.env`의 PostgreSQL 값을 반드시 채울 것.
> (기본값 폴백이 필요하면 `os.environ.get("KEY") or "기본값"` 패턴 사용.)

## 4. 가상환경(venv) 컴포넌트별 관리

### 4.1 원칙
- **컴포넌트마다 독립 `.venv`**를 둔다 (`django/.venv`, `fastapi_app/.venv`, `worker/.venv`).
- 각 컴포넌트는 **자기 `requirements.txt`**만 관리 → 의존성 충돌 차단, Docker 이미지 분리와 정합.
- `.gitignore`의 `.venv/` 패턴이 **중첩 경로까지 모두** 무시(예: `django/.venv/`).

### 4.2 컴포넌트별 셋업 명령 (Windows PowerShell 기준)

```powershell
# Django
cd django
python -m venv .venv
.\.venv\Scripts\activate
pip install -r requirements.txt
deactivate

# FastAPI
cd ..\fastapi_app
python -m venv .venv
.\.venv\Scripts\activate
pip install -r requirements.txt
deactivate

# Worker
cd ..\worker
python -m venv .venv
.\.venv\Scripts\activate
pip install -r requirements.txt
```

> Git Bash / macOS / Linux: `source .venv/bin/activate`

### 4.3 실행 예시 (로컬)
```powershell
# Django (django/ 에서)
.\.venv\Scripts\python.exe manage.py runserver

# FastAPI (fastapi_app/ 에서)
.\.venv\Scripts\python.exe -m uvicorn main:app --reload --port 8001

# Worker (worker/ 에서)
.\.venv\Scripts\python.exe main.py
```

### 4.4 Python 버전 메모
- 사용자 코드 실행(Worker sandbox)은 **Python 3.11** 기준(지시문 §3).
- 로컬 개발 파이썬은 3.13이라 `psycopg2-binary`(3.13 휠 없음) 대신 **`psycopg[binary]`(psycopg3)** 사용.

## 5. 경로 변경(레포 루트 → `django/`)에 따른 점검 결과

| 항목 | 상태 | 비고 |
|---|---|---|
| `.env` 로딩 | ✅ 수정 완료 | `BASE_DIR.parent / ".env"` → 루트 정확히 로드 |
| `TEMPLATES.DIRS` (`BASE_DIR/templates`) | ✅ 유효 | templates가 `django/`로 함께 이동 |
| `STATICFILES_DIRS` (`BASE_DIR/static`) | ✅ 유효 | static이 `django/`로 함께 이동 |
| `STATIC_ROOT` (`BASE_DIR/staticfiles`) | ✅ 유효 | BASE_DIR 상대 경로 |
| `DJANGO_SETTINGS_MODULE=config.settings` | ✅ 유효 | manage.py가 `django/`에서 실행 |
| `manage.py check` | ✅ 0 issues | 새 위치에서 정상 |

**결론**: BASE_DIR 기준 경로들은 관련 폴더가 함께 이동해 수정이 불필요했고,
레포 경계를 넘던 `.env`만 조정하면 되며 이미 반영됨. **기존 Django 코드 추가 수정 없음.**

## 6. 주의사항 / 컨벤션
- **FastAPI 폴더명은 `fastapi_app`** 권장. `fastapi`로 두면 루트에서 실행 시
  `import fastapi`가 로컬 폴더를 먼저 잡을 위험(shadowing)이 있어 지시문도 `fastapi_app`을 사용.
- `.env`는 **절대 커밋 금지**(비밀키 포함). 팀 공유는 `.env.example` 갱신으로.
- 컴포넌트 간 내부 호출은 `X-Internal-API-Key` / `X-Request-ID` 헤더 사용(지시문 규칙).
- 새 컴포넌트 추가 시 체크리스트: ① 폴더 생성 ② `.venv` + `requirements.txt` ③ 루트 `.env` 로드 코드 ④ docker-compose 서비스 등록.
