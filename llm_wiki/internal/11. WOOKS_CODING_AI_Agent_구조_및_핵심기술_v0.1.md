## 0. 한 줄 요약

WOOK'S CODING의 AI는 **5개 기능 Agent**(테스트케이스 생성 · coding_state · 힌트 · 오답노트+RAG · 미니튜터)로 구성되며,
이들은 **`CodingState`(AI가 요약한 사용자 학습 프로필)** 라는 공유 상태와 **오답노트 RAG(ChromaDB)** 를 매개로 유기적으로 협업한다.
단, **"Agent = LangGraph 자율 에이전트"는 아니다.** 실제로는 **Django가 절차적으로 오케스트레이션하고 FastAPI가 stateless LLM/RAG 도구를 제공**하는 구조다(§2 참고).

## 1. 아키텍처 한눈에

```
                          ┌─────────────────────────── FastAPI (/ai/*, stateless) ───────────────────────────┐
 사용자 ─Nginx─▶ Django ──┤  hint · wrong-note(search/analyze/embed/ask/report) · coding-state · tutor        │
        (웹/인증/권한/DB) │  · authoring(generate/fix) · lab · health/diagnostics                              │
                          │        │                         │                                                 │
                          │     OpenAI                    ChromaDB (벡터, FastAPI 소유)                         │
                          └────────┼─────────────────────────┼─────────────────────────────────────────────┘
        PostgreSQL(원본, Django 소유)  ←── 집계/저장 ──┘   wrong_note_embeddings-<embedder-sig>
```

- **호출은 단방향**: `사용자→Nginx→Django(/ai-proxy/*)→FastAPI(/ai/*)→(Chroma/OpenAI)`. FastAPI→Django 역호출 없음(확인됨).
- **데이터 소유권(SSOT 준수 확인)**: Django=PostgreSQL 쓰기 주인, FastAPI=ChromaDB 쓰기 주인 + **완전 stateless**(모든 입력을 payload로 수신, `psycopg` import 전무). Worker=코드 실행.
- **공유 상태 = `CodingState`**: 제출/오답노트에서 산출한 사용자 학습 프로필. 힌트·튜터·오답분석에 프롬프트 컨텍스트로 주입.

### 5개 Agent 요약표

| Agent | FastAPI 엔드포인트 | 실행 함수 | LLM | RAG | 성격 |
|---|---|---|---|:--:|---|
| 테스트케이스 생성 | `/ai/authoring/generate`,`/fix` | `services/authoring.py` | gpt-4o-mini `T=0.2` | ✗ | 생성+Worker실행+디버그루프 (Django 오케스트레이션) |
| coding_state | `/ai/coding-state/summarize` | `services/coding_state.py` | gpt-4o-mini `T=0.3` | ✗ | 결정적 집계 + 1-shot LLM 요약 |
| 힌트 | `/ai/hint` | `llm.generate_hint` | gpt-4o-mini `T=0.3` | ✗ | 레벨 1~3 소크라테스식, coding_state 개인화 |
| 오답노트+RAG | `/ai/wrong-note/{search,analyze,embed,ask,report}` | `llm.*`+`services/chroma.py` | gpt-4o-mini `T=0.3` | ✓ | 6섹션 분석 + 2단계 RAG 리포트 + 취약점 회수 |
| 미니튜터 | `/ai/tutor/chat` | `llm.tutor_reply` | gpt-4o-mini `T=0.5` | ✓ | 멀티턴 · 3중 개인화(상태+최근오답+RAG) |

## 2. ⚠️ "Agent"라는 표현의 정확한 정의 (설명 vs 실제)

프로젝트 문서·구두 설명에서 "LangGraph Agent"로 부르지만, **코드상 LangGraph 런타임은 존재하지 않는다.**

- `fastapi_app/agents/langgraph_agents.py`는 23줄짜리 **스텁**이다. `build_hint_agent()`·`build_analyze_agent()`·`build_ask_agent()` 전부 `raise NotImplementedError("STEP-06에서 구현")`. `StateGraph`·노드·엣지 **없음**. `main.py`에서 import조차 안 됨.
- `services/prompts.py`의 프롬프트 스텁(`HINT_SYSTEM` 등)도 **어디서도 import되지 않는다.** 실제 사용 프롬프트는 전부 `services/llm.py`·`services/authoring.py`·`services/coding_state.py`·`services/wrong_note_report.py`에 하드코딩되어 있다.
- 즉 각 "Agent"는 **`async def` 서비스 함수가 OpenAI `chat.completions.create`를 직접 호출**하는 형태다. 다단계 흐름(테스트케이스 생성의 생성→실행→디버그, 오답노트의 2단계 RAG)은 **LangGraph가 아니라 Django/FastAPI의 절차적 코드**로 구현되어 있다.

> **정확한 표현**: "LangGraph 기반 멀티에이전트"가 아니라 **"프롬프트 + RAG 오케스트레이션 파이프라인"**. 문서 7 §11.2·§11.3도 "LangGraph를 쓰지 않을 경우 미사용 사유를 남기라"고 예고했고, 실제로 그 경로를 택했다.
> (참고: 문서 7 §11.2는 LLM 생성층도 "의도적 미구현"이라 했으나, **그 뒤 실제 구현되어 운영에서 OpenAI 호출이 정상 동작 중**이다. `/ai/health`가 아직 `llm_status:"not_implemented"`를 반환하는 것은 하드코딩된 옛 문자열일 뿐이다.)

## 3. 공통 기반 (Core)

### 3.1 LLM / 임베딩 코어
- **모델**: `config.OPENAI_MODEL`(기본 `gpt-4o-mini`). 키 `OPENAI_API_KEY` 미설정 시 생성은 `not_implemented`/`failed`로 실패(정상 폴백 없음).
- **호출 패턴**: 대부분 `response_format={"type":"json_object"}` + 낮은 temperature(0.2~0.5)의 단일 호출. 스트리밍 없음.
- **임베딩**: `text-embedding-3-small`(`EMBED_MODEL`), 키 없으면 **결정적 해시 BoW 256-dim 폴백**(`RAG_EMBED_BACKEND=auto`). ← 생성엔 폴백 없지만 임베딩엔 오프라인 폴백 있음.

### 3.2 보안 · 전송
- Django `ai_proxy.client.call_fastapi`가 `urllib`으로 `FASTAPI_BASE_URL`(기본 `http://fastapi:8001`)에 POST.
- 헤더: `X-Internal-API-Key`(불일치 401) + `X-Request-ID`(없으면 400). FastAPI는 `Depends(verify_internal)`로 검증 — **`/ai/health` 제외 전 엔드포인트 적용**.
- 타임아웃: 기본 20s, coding-state/tutor는 90s, 테스트케이스 생성은 120s.
- **Stub 가드**: 응답에 `"[stub]"` 문자열이 있으면 Django가 `failed/not_implemented`로 강등.

### 3.3 로깅 (2-plane)
- **DB 감사**: Django가 `LLMRequestLog`(요청/응답/상태) + 실패 시 `ErrorLog`를 PostgreSQL에 저장.
- **AI 연구 로그**: FastAPI `log_ai_event()`가 `volumes/logs/ai/research.jsonl`에 JSON Lines 기록(회전 5MB×5, `app.log`와 분리). 계측 이벤트: `hint`·`tutor_chat`·`rag_search`·`analyze`·`rag_embed`·`note_ask`·`wrong_note_report`.
  - **미계측**: `coding-state/summarize`, `authoring/*`, `lab/*`, `diagnostics` (활성 경로인데 research.jsonl엔 안 남음 — 개선 여지).

## 4. 각 Agent 상세

### 4.1 테스트케이스 생성 Agent

문제에 TC가 없을 때 **정답코드·입력을 자동 생성**한다. **오케스트레이션은 Django(`problems/services/testcase_agent.py`), 생성은 FastAPI(`/ai/authoring/*`)** 로 분리.

파이프라인(`generate_testcases`):
1. **생성**(`/ai/authoring/generate`, `T=0.2`): LLM이 `solution_code`(정답), `generator_code`(seed→랜덤입력 1개 출력), `edge_inputs`(1~5), `time_inputs`(1~5), `notes`를 JSON으로 반환. 프롬프트가 **넓은 값 범위·중복 회피**를 명시.
2. **랜덤 입력 생성**: `generator_code`를 seed `0..29`(기본 30)로 **Worker `code_eval` 잡**에 위임 실행 → 정상 종료한 stdout만 수집.
3. **정답 실행 + 크래시→수정 루프**(`_run_solution`, `max_fix=1`⇒최대 2회): 후보 입력(edge+time+random)을 정답코드로 실행. 크래시(`returncode∉{0,None}`) 시 `/ai/authoring/fix`에 stderr(≤1500자)·입력(≤1000자)을 보내 정답코드를 수정 후 재시도.
4. **기대출력 산출**: 정상 종료한 케이스만 채택, 정답코드 stdout=`expected_output`, `compare_mode="line_trim"` 고정.
5. **저장**(`generate_and_save`): `problems.TestCase`로 bulk_create, **edge 앞 2개는 공개 샘플(`is_sample=True`)**.

- Worker 위임: `_eval_batch`가 `ExecutionJob(job_type="code_eval")` 생성 후 1초 간격 폴링(최대 `poll=180s`).
- **역할 분리**: FastAPI=stateless LLM 도구 2개(`generate`/`fix`), Django=루프·크래시 판정·Worker 위임·DB 저장.

### 4.2 coding_state Agent — 사용자 학습 상태의 핵심

**사용자별 1행(`OneToOne`) 학습 프로필**을 유지한다. Django `codingstate` 앱의 `CodingState` 모델에 저장.

**저장 필드(`CodingState`)**:

| 필드 | 의미 |
|---|---|
| `summary` | 2~4문장 진단 요약(prose) |
| `level` | 추정 수준 `입문/초급/중급/고급` |
| `strengths` / `weaknesses` | 강한/약한 알고리즘·유형(LLM 추론, JSON list) |
| `recurring_mistakes` | 반복 실수 패턴 |
| `recommended_focus` | 다음에 집중할 주제 |
| `stats_snapshot` | 이번 갱신에 쓰인 결정적 집계 원본(감사/diff용) |
| `source_submission_count` | 갱신 시점 제출 수(staleness 게이트) |
| `model` | 사용 LLM id |

**생성 흐름**:
1. **결정적 집계**(Django `gather_stats`, LLM 없음): `Submission`(정답률·난이도별·최근10건), `ProblemTag`(푼/틀린 태그 top10), `WrongNote`(반복 오류패턴 top8·오답수·복습수) → `stats` dict.
2. **LLM 요약**(FastAPI `/ai/coding-state/summarize`, `gpt-4o-mini T=0.3` JSON): stats를 받아 level/strengths/weaknesses/recurring_mistakes/recommended_focus/summary를 **한국어로** 산출. 시스템 프롬프트가 "이 결과는 **내부용, 사용자에게 절대 노출 금지**, 솔직하게 진단"이라고 지시.
3. **저장**: `update_or_create`로 CodingState 갱신.

**갱신 트리거(중요)**: `ensure_fresh(user, min_new_submissions=5)` — 상태가 없으면 생성, 있으면 **직전 갱신 이후 제출이 5건 이상 늘었을 때만** 재요약(LLM 비용 절감). 실제 자동 트리거는 **오답노트 완료 시점 1곳**(`wrongnotes/views.py`)뿐. 그 외엔 관리 명령 `refresh_coding_state`.

**소비**: `get_prompt_context(user)`가 `[사용자 코딩 상태 — AI 참고용, 노출 금지]` 라벨 텍스트로 렌더 → **힌트·오답분석·튜터**의 payload `coding_state` 키로 주입(§5).

> 성격: **결정적 집계 + 1-shot LLM 요약**. 실질적 LLM 추론은 있으나 tool-use/멀티스텝/자기수정 없는 비(非)에이전트. 정확한 표현은 *"AI가 요약한 내부 학습자 프로필"*.

### 4.3 힌트 Agent

`/ai/hint` — 정답을 주지 않는 **소크라테스식 단계 힌트**.

- **요청**: 프론트는 `{problem_id, hint_level, user_code}`만 전송. **Django view가 문제 지문·`level`·`coding_state`를 서버측에서 enrich**.
- **프롬프트 주입**: 문제(제목/난이도/설명/제약) + 사용자 코드 + `hint_level`(1~3) + `level`(추정 수준) + `coding_state`. 시스템 프롬프트가 **수준별 스타일 분기**와 **레벨별 깊이 분기**(1=개념 오리엔테이션, 2=구체 과정, 3=최대 스캐폴딩) 지시. 정답/전체 접근법 코드 노출 금지.
- **모델**: `gpt-4o-mini T=0.3`, JSON `{content}`, 스트리밍 없음.
- **다단계**: 레벨 1~3, 클라이언트가 상태 관리(서버 stateless). 프론트에서 "다음 힌트"로 1→2→3 순차 해금.
- **개인화**: coding_state 주입은 실제 배선됨. 단 **오답 약점은 CodingState 요약을 경유한 간접 주입**(특정 오답노트 원문은 안 들어감).

### 4.4 오답노트 Agent + RAG — 취약점 분석의 중심

`/ai/wrong-note/{search, analyze, embed, ask, report}` 5개 엔드포인트.

#### (a) AI 분석 `/analyze` (`llm.analyze_wrong_note`, `gpt-4o-mini T=0.3` JSON)
**6개 섹션** 출력: `problem_core`(문제 핵심) · `solution`(정석 접근) · `cause`(오답 원인) · `improvement`(개선) · `ai_feedback`(자유 피드백) · `next_checklist`(2~4개, "자주 하는 실수" 체크리스트). `coding_state`를 주입받아 학습자 수준·약점에 맞춰 진단.

#### (b) RAG WRITE — 인덱싱/임베딩 `/embed` (`chroma.embed_note_chunked`)
- **컬렉션(임베더 버전관리)**: `wrong_note_embeddings-<sig>`. OpenAI 사용 시 `wrong_note_embeddings-oa-text-embedding-3-small`, 해시 폴백 시 `wrong_note_embeddings-hash256`. cosine 공간(`hnsw:space:cosine`). → 임베더가 바뀌어도 차원 충돌 없이 격리.
- **청킹 전략(`build_note_chunks`+`chunk_text`)**:
  - 인덱싱 대상 = **회고 + AI 분석 섹션 텍스트만** + `topic` 청크 1개(`카테고리 · 알고리즘 태그`). **문제 지문·제출 코드·난이도·에러·결과는 제외**(노이즈 배제).
  - `chunk_text`: 문장 경계로 나눈 뒤 **≤160자(`RAG_CHUNK_MAX_CHARS`)** 로 팩킹(긴 문장은 하드 분할). → 노트당 `topic 1` + 섹션별 N개(대략 6~8 청크/노트).
  - **메타데이터**: `user_id, wrong_note_id, source, section, chunk_index, indexed_at, problem_title, category, algo_tags`.
  - 재인덱싱 시 `delete(where={wrong_note_id})`로 **이전 청크(구 v0 단일벡터 포함) 제거 후 upsert**.

#### (c) RAG READ — 리트리빙 `/search`·`search_user_notes` (v1 전략)
- 쿼리도 `chunk_text`로 **멀티 청크화** → 각 쿼리 청크로 `n_results=12`(`RAG_PER_QUERY_K`) 조회, **`where={"user_id"}` 격리 필수**.
- **(노트, 섹션)별 최고 유사도만 유지**(`sim = 1 - cosine_distance`, 자기 노트/노이즈 제외).
- **노트 단위 집계(`aggregate_section_sims`)**:
  ```
  score = (RAG_W_MEAN·mean(section_sims) + RAG_W_MAX·max(section_sims)) / (W_MEAN+W_MAX)
        = 0.5·mean + 0.5·max         # 기본 가중치 0.5/0.5
  ```
  `mean`=여러 섹션에 두루 유사한 노트 보상, `max`=한 섹션이 강하게 일치하는 노트 보상.
- **필터/정렬**: `score ≥ RAG_SCORE_THRESHOLD(0.35)` 만 남기고 내림차순, `top_k=5`. 반환은 노트 단위 `Evidence{note_id, source, score, title}`(청크는 노트로 접힘).

#### (d) 2단계 RAG 리포트 `/report` (`wrong_note_report.generate_report`)
1. **1차 검색(증상)**: `q1 = comment`(없으면 `제목+태그`) → `search_user_notes` → `similar1`(자기 제외).
2. **AI 코멘트 생성**(`_llm_report`, JSON): `retrospection_feedback, missed_points[], learning_direction[], summary{problem_core,cause}`.
3. **2차 검색(진단 보강)**: `enriched = comment + missed_points + learning_direction + cause` → 재검색 → `similar2`.
   → 증상 기반 검색 후, LLM 진단으로 쿼리를 보강해 다시 검색하는 **증상→진단 2-hop RAG**.

#### (e) 취약점 분석 방식
- **전용 통계 롤업 엔드포인트는 없음.** 취약점은 3경로로 표현:
  1. **노트별**: `cause`·`next_checklist`·`error_pattern`(WrongNote 필드).
  2. **집계 요약**: `gather_stats`가 `WrongNote.error_pattern`을 top8로 집계 → CodingState의 `weaknesses`/`recurring_mistakes`로 승화 → 힌트·튜터·분석에 재주입.
  3. **동일 오류군 회수(RAG)**: 리트리빙이 "같은 오류 유형 노트"를 잘 찾도록 최적화(§4.4c, §7).

#### (f) v0→v1 성능(문서 10 · `rag_eval.py`)
| | v0(단일 blob 벡터) | v1(섹션 멀티청킹+mean/max) |
|---|---|---|
| 인덱스 | 노트=1 벡터(코드·지문 포함) | 노트=N 섹션 청크(회고+AI섹션만) |
| 스코어 | 단일 cosine | 섹션 max→mean·max 가중 |
| **Precision@4** | 0.625 | **1.000** |
| **Recall@4** | 0.625 | **1.000** |
| **MRR / nDCG@4** | 0.742 / 0.618 | **1.000 / 1.000** |

- **핵심 교훈**: v1 구조만으론 부족. **해시 BoW 임베딩에선 v1 Precision이 0.24~0.54로 오히려 악화**(짧은 한국어 청크가 공통 조사/저차원 해시 충돌에 지배됨). **OpenAI 시맨틱 임베딩(text-embedding-3-small)** 과 결합해야 짧은 청크 분리가 정밀도로 전환됨(1.00). → `RAG_EMBED_BACKEND=auto`(키 있으면 OpenAI).

### 4.5 미니튜터 Agent

`/ai/tutor/chat` — **멀티턴 대화형** 튜터. 가장 "유기적"인 Agent.

- **3중 개인화 컨텍스트**(Django가 조립, `_tutor_context_block`으로 system 보조 메시지화):
  1. `activity` — 현재 페이지/풀고 있는 문제(#id·제목·난이도·태그).
  2. `coding_state` — `get_prompt_context`(힌트와 동일한 상태 요약).
  3. `recent_notes` — **최근 30일 WrongNote 최대 8건**(error_pattern·경과일·요약) **원본 직접 주입**.
  4. `evidence`(RAG) — FastAPI가 질문으로 `chroma.search_user_notes` 실행, 실패해도 대화 진행. 응답 `used_notes`로도 반환.
- **모델**: `gpt-4o-mini T=0.5, max_tokens=500`, **평문 응답**(JSON 아님), 스트리밍 없음.
- **멀티턴**: 이력은 **클라이언트 보존**(프론트 8턴 전송 → Django 정제 → FastAPI 최근 6턴만 messages 반영). 서버 stateless.

## 5. 유기적 협업 구조

**공유 substrate = `CodingState`.** 활동 데이터가 상태로 응축되고, 그 상태가 다시 각 활동의 프롬프트로 되돌아가는 **피드백 루프**가 실제로 배선되어 있다.

```
   제출/오답노트 ──gather_stats(집계)──▶ /ai/coding-state/summarize ──▶ CodingState
        ▲                                                                │
        │                                              get_prompt_context│(coding_state 텍스트)
        │                                                                ▼
   오답노트 작성 ──/analyze(6섹션)──▶ ai_analysis ──build_sections──▶ /embed ──▶ ChromaDB
        │                                                                │           │
        │                                                        (search_user_notes) │
        └────────────────────────── 힌트 · 튜터 · 오답분석 ◀──────────────┴───(RAG)──┘
                                     (coding_state 주입 + 튜터는 최근오답+RAG 추가)
```

**실제로 동작하는 루프**:
1. **제출·오답 → coding_state → 프롬프트 재주입**: 힌트(`coding_state`+`level`), 오답분석(`coding_state`), 튜터(`coding_state`)에 주입.
2. **분석 → 임베딩 → RAG**: `/analyze` 산출 섹션을 `/embed`로 인덱싱 → `/search`·`/ask`·튜터 RAG로 회수(닫힌 루프).
3. **최근 오답 → 튜터**: 최근 30일 노트 원본 + 질문기반 RAG를 튜터에 직접 주입.

## 6. "사용자 상태를 자세히 갖는다"는 것의 실체

사용자 상태는 **두 층**으로 표현된다:
- **원천 데이터(PostgreSQL)**: `Submission`(정답률·난이도별·최근활동), `WrongNote`(회고·AI분석 6섹션·`error_pattern`·복습여부), `ProblemTag`(푼/틀린 태그).
- **응축된 프로필(`CodingState`)**: 위를 `gather_stats`로 결정적 집계 → LLM이 `level·strengths·weaknesses·recurring_mistakes·recommended_focus·summary`로 진단. `stats_snapshot`에 원본 집계도 보존.

이 프로필이 힌트·튜터·오답분석의 프롬프트에 **"[내부용, 노출 금지]" 라벨과 함께 주입**되므로, 각 AI가 "이 사용자는 중급, DP 약함, 인덱스 범위 실수 반복" 같은 맥락을 갖고 응답한다. → **"모든 활동에서 활용"은 사실.**

## 7. 내 설명(구두)과 실제 구현의 차이 — 검증 결과

| # | 설명한 내용 | 판정 | 실제 |
|---|---|:--:|---|
| 1 | 5개 Agent가 유기적으로 협업 | ✅ 대체로 맞음 | CodingState+RAG를 매개로 실제 피드백 루프 존재 |
| 2 | coding_state·오답노트로 사용자 상태를 자세히 보유 | ✅ 맞음 | `CodingState` 모델(6+2 필드) + WrongNote 6섹션. 근거 확실 |
| 3 | 모든 활동에서 **활용** | ✅ 맞음 | 힌트·튜터·오답분석 모두 `coding_state` 주입 배선됨 |
| 4 | 모든 활동에서 **업데이트** | ⚠️ 뉘앙스 | 매 활동이 아니라 **오답노트 완료 시 + 제출 5건 누적 게이트**로만 갱신. 제출만으론 자동 갱신 안 됨 |
| 5 | 오답노트가 취약점 분석에 활용 | ✅ 맞음(단서) | 전용 통계 롤업은 없음. `error_pattern` 집계→CodingState weaknesses + 동일오류군 RAG로 표현 |
| 6 | RAG 청킹/리트리빙 전략 | ✅ 맞음 | 섹션 멀티청킹(≤160자)+`(0.5·mean+0.5·max)` 집계+user_id 격리. 잘 구현됨 |
| 7 | (암묵) LangGraph 기반 에이전트 | ❌ 틀림 | **LangGraph 미사용**(스텁). 직접 OpenAI 호출 + 절차적 오케스트레이션 |

**추가로 바로잡을 점(코드 실측)**:
- **힌트의 오답 약점 주입은 간접**(CodingState 요약 경유). 오답노트 원문을 직접 넣는 건 **튜터뿐**.
- **`/analyze`는 RAG 근거를 소비하지 않음**: Django가 `/search`와 `/analyze`를 **독립 호출**해 둘 다 저장할 뿐, analyze 프롬프트엔 `evidence=[]`. 유사노트가 분석에 직접 반영되진 않음.
- **`/ai/wrong-note/report`(2단계 RAG 리포트)는 Django에서 호출되지 않는 고아(orphan) 엔드포인트.** 구현은 완료됐으나 웹 경로 미배선(문서 9 §10의 "미구현" 예고와 일치).
- **`/ai/wrong-note/ask`(내 노트 질의)**: 최근 프론트에서 '내 노트에 질문' 페이지를 제거했으므로, 이를 호출하던 UI가 사라짐(ai_proxy 프록시 라우트는 잔존 가능 → 정리 대상).
- **`/ai/health`의 `llm_status:"not_implemented"`는 옛 하드코딩 문자열**. LLM 생성은 실제 동작 중.

## 9. 개선 여지 (백로그)

- `/ai/wrong-note/report` 2단계 RAG 리포트를 Django에 배선(현재 고아). — 취약점 리포트 UX의 완성.
- `/analyze`에 `/search` 결과를 실제 프롬프트 근거로 주입(현재 `evidence=[]`).

- coding-state·authoring Agent를 `research.jsonl`에 계측 추가.
- `/ai/health`의 상태 문자열을 실제 구현 상태로 갱신.
- (선택) LangGraph 실제 도입 여부 결정 — 현재는 절차적 구조로 충분히 동작하므로 "미사용" 사유를 문서에 확정.
