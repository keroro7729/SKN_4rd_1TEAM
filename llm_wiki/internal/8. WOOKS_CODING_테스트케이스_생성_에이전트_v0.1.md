# WOOK'S CODING — 테스트케이스 생성 에이전트 (v0.1)

작성일: 2026-07-06 · 대상: 문제 하나로부터 정답코드·테스트케이스를 자동 생성

> 문제 지문을 입력하면 **정답(레퍼런스) 코드 + 랜덤/엣지/시간 테스트케이스**를 자동 생성한다.
> LLM 생성(FastAPI) · 코드 실행(Worker) · 오케스트레이션(Django)의 3-컴포넌트 협업으로,
> 프로젝트 아키텍처 규칙(FastAPI=AI, Worker=실행, Django=DB/조율)을 그대로 따른다.


---

## 1. 설계 결정 (확정)

| 항목 | 결정 |
|---|---|
| 랜덤 TC 생성 방식 | **제너레이터 스크립트** — LLM이 "seed→랜덤 small 입력"을 출력하는 생성기를 만들고, 워커로 N회 실행 |
| LLM 백엔드 | **실제 OpenAI** (`openai==1.35.7`, `OPENAI_MODEL` 기본 `gpt-4o-mini`) |
| 출력 처리 | **dry-run** — 최종 TC 리스트를 반환만 하고 DB(`problems.TestCase`)에는 저장하지 않음 |

## 2. 아키텍처 / 데이터 흐름

```
Django (오케스트레이터 = Agent)
  │  ① call_fastapi(/ai/authoring/generate)
  ├─────────────────────────────▶ FastAPI (LLM)
  │                                 · 정답코드 + 제너레이터 + 엣지/시간 입력 생성
  │  ④ (크래시 시) /ai/authoring/fix   · 디버깅용 코드 수정
  │◀─────────────────────────────
  │
  │  ② 제너레이터 실행(seed 0..N-1)   ③ 정답코드 실행(엣지+시간+랜덤)
  └── ExecutionJob(code_eval) ──▶ Worker (샌드박스)
                                    · code+inputs 배치 실행 → stdout/시간 캡처 (비교 없음)
  │◀── result_payload.results ──
  │
  ⑤ 최종 TC 리스트(random/edge/time) 반환
```

- **책임 분리 유지**: FastAPI는 RDB 미접근(payload in/out), Worker는 DB 잡 폴링으로 실행만, Django가 DB·조율 소유.

## 3. 컴포넌트별 구현

### 3.1 FastAPI — LLM 생성
- `fastapi_app/services/authoring.py` — 실제 `AsyncOpenAI` 클라이언트(지연 초기화).
  - `generate(problem) -> {solution_code, generator_code, edge_inputs[], time_inputs[], notes}`
  - `fix(problem, solution_code, error, sample_input) -> {solution_code, notes}`
  - `response_format={"type":"json_object"}`, `temperature=0.2`. 실패 시 `AuthoringError`.
- `fastapi_app/routers/authoring.py` — `POST /ai/authoring/generate`, `POST /ai/authoring/fix`
  - `Depends(verify_internal)` (X-Internal-API-Key / X-Request-ID)
  - 성공: `status=success` + 필드, 실패: `status=failed` + `message`
- `fastapi_app/schemas/authoring.py`, `main.py`에 라우터 등록.

**프롬프트 규약(중요):** 모든 프로그램은 self-contained Python 3, **stdin→stdout**.
- `generator_code`: 첫 줄에서 정수 `seed`를 읽어 `random.seed(seed)` 후 **유효한 small 입력 1개**를 출력.
- `solution_code`와 `generator_code`는 **입력 포맷이 반드시 일치**해야 함.

### 3.2 Worker — 신규 `job_type='code_eval'`
- `worker/main.py`
  - 폴링 `IN (...)`에 `code_eval` 추가, 클레임 SELECT에 `input_payload` 포함.
  - `process_eval(conn, job_id, input_payload)`: `{code, inputs[], timeout}`를 **배치 실행**.
    - 각 입력마다 `run_code(code, stdin)` → 결과 수집 (비교/채점 없음).
    - `result_payload = {kind:"code_eval", count, results:[{stdout, stderr, returncode, timed_out, elapsed_ms}, ...]}`
  - 사용자/제출과 무관한 **시스템 잡** → `user`/`submission` 없이 동작.
- 기존 하드닝된 `runner.run_code`(메모리/CPU/파일쓰기 rlimit + 네트워크 차단) 재사용.
- **배치 설계 이유**: 생성기 100회·정답코드 N회를 각각 1개 잡으로 처리 → 폴링 큐 왕복 최소화.

### 3.3 Django — 오케스트레이터 (관리 명령)
- `problems/management/commands/generate_testcases.py`
- `ExecutionJob.user`/`submission`을 **nullable**로 변경(마이그레이션 `submissions/0003_*`) — code_eval 시스템 잡용.
- choices 추가: `JOB_TYPE_CHOICES += code_eval`, `LLM_REQUEST_TYPE_CHOICES += testcase_gen, testcase_fix`.

**실행 흐름:**
1. `[1/4]` FastAPI `generate` → 정답코드·제너레이터·엣지/시간 입력
2. `[2/4]` 워커 `code_eval`로 제너레이터를 `seed 0..N-1` 실행 → 랜덤 small 입력 N개(성공/비공백만 채택)
3. `[3/4]` 워커 `code_eval`로 정답코드를 (엣지+시간+랜덤) 입력에 실행 → 기대출력·실행시간
   - **디버깅 루프**: 정답코드가 크래시(returncode≠0)하면 FastAPI `fix` 호출 후 재실행 (`--max-fix`회)
   - 타임아웃 케이스는 해당 케이스만 제외(크래시가 아니면 수정 대상 아님)
4. `[4/4]` 성공 케이스만 최종 TC로 조립 후 **JSON 반환**(dry-run)

## 4. 실행 방법

```bash
docker compose -f docker-compose.yml -f docker-compose.local.yml \
  run --rm django python manage.py generate_testcases <problem_id> [옵션]
```

전제: `postgres`·`worker`·`fastapi` 기동 상태 + `.env`의 `OPENAI_API_KEY` 유효.

| 옵션 | 기본값 | 설명 |
|---|---|---|
| `problem_id` | (필수) | 대상 문제 PK |
| `--seeds` | `100` | 랜덤 TC 개수 |
| `--timeout` | `5` | 입력당 실행 제한(초) |
| `--max-fix` | `2` | 디버깅 루프 최대 횟수 |
| `--poll` | `180` | 워커 잡 폴링 제한(초) |
| `--out` | (없음) | 결과 JSON 저장 경로(미지정 시 stdout) |

## 5. 입출력 계약

**FastAPI `/ai/authoring/generate` 응답**
```json
{"status":"success","solution_code":"...","generator_code":"...",
 "edge_inputs":["..."],"time_inputs":["..."],"notes":"..."}
```

**Worker `code_eval` 잡**
- `ExecutionJob.input_payload = {"code": "...", "inputs": ["...", ...], "timeout": 5}`
- `ExecutionJob.result_payload = {"kind":"code_eval","count":N,
   "results":[{"stdout","stderr","returncode","timed_out","elapsed_ms"}, ...]}`

**명령 최종 출력(dry-run)**
```json
{
  "problem_id": 1978,
  "counts": {"random": 100, "edge": 5, "time": 5, "total": 110},
  "test_cases": [
    {"kind":"edge","input":"1 1","expected_output":"2\n","elapsed_ms":24,"compare_mode":"line_trim"}
  ]
}
```
- `kind` ∈ `random | edge | time`.

## 6. 검증 결과 (2026-07-06)

- ✅ **워커 code_eval 결정적 검증**: 생성기(seed→랜덤입력) + 정답코드(a+b) 주입 → 입력별 정확 산출(합 검증 True).
- ✅ **실제 OpenAI 전체 파이프라인**: 문제 "두 수의 합"(#1978, `--seeds 20`)
  → `solution 74B · generator 151B · edge 5 · time 5` → 랜덤 20/20 → 최종 **random 20 · edge 5 · time 5**.
  엣지 `1 1→2`, `1000 1000→2000`, `500 500→1000` 등 전부 정답.
- ✅ `manage.py check` 0 issues, 마이그레이션 0003 정상 적용.

## 7. 한계 / 백로그
- **디버깅 루프**: 코드 경로는 구현·검증됐으나, 정상 정답코드에서는 미발동(버그 있는 레퍼런스에서만 트리거).
- **DB 저장 미구현(의도)**: 현재 dry-run. 향후 `--save` 옵션으로 `problems.TestCase` 행 저장(카테고리 구분 필드 필요) 확장 여지.
- **품질 편차**: LLM 특성상 애매한 지문에선 생성기/정답코드 입력포맷 불일치 가능 → 디버깅 루프/재시도로 완화.
- **누적 잡 정리**: 대량 `code_eval` ExecutionJob 행이 쌓임(무해, 필요 시 정리).
- **운영 편의**: 관리자 화면에서 문제별 "에이전트 실행" 버튼, 배치 대상 선택 등은 후속 과제.
