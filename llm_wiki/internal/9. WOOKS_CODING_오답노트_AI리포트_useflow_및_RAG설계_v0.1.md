# WOOK'S CODING — 오답노트 AI 리포트 useflow & RAG 설계 (v0.1)

작성일: 2026-07-06 · 기준: 현재 코드베이스 실측(`django/wrongnotes/*`, `fastapi_app/*`, ChromaDB)

> 목적: 오답노트를 **"작성"과 "AI 리포트 생성"으로 분리**하는 개편(useflow)의 기능 요구사항,
> 2단계 RAG **Agent flow**, 데이터 모델/테이블 변경, ChromaDB **RAG 문서 구조 1차 초안**을 규정한다.

## 1. 현재(as-is) 오답노트 도메인

### 1.1 모델 · 테이블 (`django/wrongnotes/models.py`)

| 모델 (테이블) | 필드 | 비고 |
|---|---|---|
| **WrongNote** (`wrongnotes_wrongnote`) | user FK, problem FK, submission FK, tags M2M(ProblemTag), `comment`(사용자 회고), `ai_analysis`(JSON), `error_pattern`(≤100), `status`, `is_reviewed`, `reviewed_at`, `created_at` | index(user,status,created_at) |
| **WrongNoteVector** (`wrongnotes_wrongnotevector`) | user FK, wrong_note **O2O**, `embedding_id`, `source`, `indexed_at` | ChromaDB 인덱싱 메타(원문 1:1) |
| **WrongNoteQueryLog** (`wrongnotes_wrongnotequerylog`) | user FK, `query`, `answer`, `evidence_note_ids`(JSON), `scores`(JSON), `created_at` | '내 노트에 물어보기' 로그 |

- `status` ∈ `WRONG_NOTE_STATUS_CHOICES` = `draft / completed / indexed / index_failed`.
- `ai_analysis`(JSON) 현재 구조: `{ "analysis": {problem_core,cause,solution,caution,evidence}, "similar_notes": [...], "errors": [...], "index": {...} }`.

### 1.2 현재 흐름 (결합형) — `WrongNoteCreateView.post`
```
[작성 POST 1회] 노트 생성/갱신(status=completed)
   → analyze_wrong_note(): /ai/wrong-note/search(유사검색) + /ai/wrong-note/analyze(AI분석)
   → ai_analysis 저장
   → embed_wrong_note(): /ai/wrong-note/embed(ChromaDB 인덱싱) → status=indexed(실패 시 index_failed)
   → record_user_action("wrongnote_completed")  # 포인트
```
**문제점**: 작성과 AI 분석/인덱싱이 한 요청에 묶여 있어 (a) 사용자가 회고를 다듬고 리포트를 "따로/다시" 받기 어렵고, (b) LLM 지연/실패가 저장 자체를 지연시킨다. 현재 `/ai/wrong-note/analyze`의 LLM은 스텁(not_implemented).

### 1.3 현재 RAG 구조 (`fastapi_app/services/chroma.py`)
- 컬렉션: **`wrong_note_embeddings`** (cosine).
- 임베딩: **결정적 hash 기반 64-dim** (MVP 검증용, 실제 임베딩 모델 미연결) — `_embed_text`.
- 인덱싱 문서(text): 문제명·난이도·태그·사용자 코멘트·제출결과·오류메시지·제출코드·AI분석(핵심/원인/풀이/주의) — `build_wrong_note_index_content`.
- 메타데이터: `user_id, wrong_note_id, source, indexed_at, problem_title`.
- 검색: **user_id 필터 필수**, `score = 1 - distance`, threshold `RAG_SCORE_THRESHOLD=0.35`, `top_k=5`.
- FastAPI: `/ai/wrong-note/{search, analyze, embed, ask}`.

## 2. 변경 개요 (as-is → to-be)

| 구분 | as-is | **to-be** |
|---|---|---|
| 작성 | 작성 = 저장 + AI분석 + 인덱싱(한 요청) | **작성 = 저장만**(사용자 회고 기록) |
| AI 리포트 | 작성 시 자동 | **"AI 리포트" 버튼으로 on-demand**(재생성 가능) |
| 유사 노트 | 1회 검색(analyze 내부) | **2단계 리트리빙**(회고기반 1차 → 전체기반 최종) |
| 리포트 내용 | problem_core/cause/solution/caution | **풀이·회고 코멘트 / 놓친 부분 / 학습 방향** + 유사 오답 패턴 |
| 인덱싱 시점 | 작성 시 | **리포트 생성 후**(리포트 포함 문서로 인덱싱) |

## 3. 기능 요구사항 (Functional Requirements)

### FR-1 오답노트 작성 (AI와 분리)
- FR-1.1 사용자는 **최종 제출(submit)** 중 `wrong/error/timeout` 결과에 대해서만 오답노트를 작성할 수 있다(현행 유지).
- FR-1.2 작성 화면은 **문제 · 제출 코드 · 실행 결과(결과/시간/에러메시지)** 를 보여주고, 사용자가 **회고 코멘트(comment)** 를 직접 입력한다.
- FR-1.3 저장 시 **AI 호출·인덱싱을 하지 않는다.** 노트는 `status=completed`(리포트 없음)로 저장. 저장은 LLM 지연과 무관하게 즉시 성공해야 한다.
- FR-1.4 저장 시 `wrongnote_completed` 포인트를 1회 지급(현행 유지, 중복 금지).

### FR-2 AI 리포트 생성 (버튼, on-demand)
- FR-2.1 노트 상세 화면의 **"AI 리포트" 버튼** 클릭 시 리포트 생성을 요청한다(비동기 UI: loading→ready/failed).
- FR-2.2 리포트는 다음을 제시한다:
  - **풀이/회고 코멘트**(retrospection_feedback): 사용자의 회고에 대한 피드백
  - **놓친 부분**(missed_points): 코드/접근에서 빠뜨린 지점
  - **학습 방향**(learning_direction): 다음에 학습/연습할 개념·유형
  - (선택) 문제 핵심·오답 원인 요약
- FR-2.3 리포트에는 **과거 유사 오답 패턴(유사 오답노트)** 이 함께 리트리빙되어 제시된다(note_id · problem_title · score · 요약).
- FR-2.4 리포트는 **재생성 가능**해야 하며(회고 수정 후 다시 요청), 최신 리포트를 노트에 연결한다.
- FR-2.5 리포트 생성 실패(LLM/네트워크) 시 노트 저장 상태에는 영향이 없어야 한다(리포트만 `failed`).

### FR-3 인덱싱 (리포트 반영)
- FR-3.1 리포트 생성 성공 후, **회고 + 리포트를 포함한 문서**를 ChromaDB에 인덱싱한다(`status=indexed`, 실패 시 `index_failed`).
- FR-3.2 인덱싱은 **user_id 스코프** 를 지키며, 재생성 시 동일 `wrong_note_id`로 upsert 한다.

### FR-4 권한/격리
- FR-4.1 모든 조회·검색은 **본인 노트만**(`user_id` 필터 필수). 다른 사용자 노트 검색 금지.

## 4. 오답노트 Agent flow (2단계 RAG)

리트리빙(ChromaDB)과 LLM이 모두 FastAPI 도메인이므로 **리포트 에이전트는 FastAPI에 위치**한다.
Django는 RDB(노트 원문/리포트 저장)를 소유하고 payload 구성·영속화·화면을 담당한다.

```
[사용자] "AI 리포트" 클릭
  │
[Django] /wrongnotes/<id>/report/  (POST)
  │   payload = {user_id, problem_title, difficulty, tags,
  │              submitted_code, result, error_message, user_comment(회고)}
  ▼
[FastAPI] /ai/wrong-note/report   (Agent)
  ① 1차 리트리빙  : chroma.search(user_id, query=회고 comment)          → similar₁
  ② AI 코멘트 생성: LLM(문제 + 제출코드 + 회고 + similar₁)
                    → {retrospection_feedback, missed_points, learning_direction, ...}
  ③ 최종 리트리빙 : chroma.search(user_id, query=회고+리포트 enriched)   → similar₂
  ④ 반환          : {status, report, stage1_evidence: similar₁, stage2_evidence: similar₂}
  ▼
[Django] WrongNoteReport 저장 + WrongNote.status 갱신
  │   evidence note_id → RDB 노트 상세로 resolve(제목/요약) 후 화면 렌더
  ▼
[Django→FastAPI] /ai/wrong-note/embed  (회고+리포트 문서 인덱싱, status=indexed)
```

- **① vs ③ 차이**: ①은 사용자의 *증상(회고)* 만으로 좁게 검색해 LLM 컨텍스트를 확보하고, ③은 LLM이 만든 *진단(리포트)* 까지 합친 enriched 쿼리로 재검색해 사용자에게 보여줄 **최종 유사 오답노트**를 고른다. (증상→진단 2-hop)
- 각 검색은 `score ≥ 0.35`, `top_k=5` 필터. 결과 없으면 `empty`(빈 근거로 리포트만 제시).
- 재생성 시 동일 흐름 반복 → 최신 리포트로 교체 + upsert 재인덱싱.

## 5. 데이터 모델 변경 1차 초안

작성/리포트 분리와 **재생성·근거 보존**을 위해 리포트를 별도 모델로 승격한다.

### 5.1 신규 — `WrongNoteReport` (`wrongnotes_wrongnotereport`)
| 필드 | 타입 | 설명 |
|---|---|---|
| wrong_note | FK(WrongNote, related_name="reports") | 1 노트 : N 리포트(재생성 이력) |
| user | FK(User) | 소유자(격리·조회 편의) |
| retro_comment | TextField | 생성 시점 회고 스냅샷 |
| retrospection_feedback | TextField | 회고 피드백 |
| missed_points | JSONField(list) | 놓친 부분 목록 |
| learning_direction | JSONField(list) | 학습 방향 목록 |
| summary | JSONField(dict) | (선택) problem_core/cause 등 요약 |
| stage1_evidence | JSONField(list) | 1차 리트리빙 근거 `[{note_id,score,title}]` |
| stage2_evidence | JSONField(list) | 최종 리트리빙 근거 |
| status | Char | `generating / ready / failed` |
| model | Char | 사용 LLM 모델명 |
| request_id | Char | 추적용(X-Request-ID) |
| created_at | DateTime | 생성일 |

> 대안(경량): 별도 모델 없이 `WrongNote.ai_analysis`(JSON)에 리포트를 계속 담고 `report_status` 필드만 추가.
> → 재생성 이력/근거 스냅샷이 필요 없으면 이 대안으로 시작 가능. 본 문서는 이력 보존을 위해 별도 모델을 권장.

### 5.2 `WrongNote` 변경
- `status` 의미 재정의: `completed`(작성 완료·리포트 없음) → 리포트 성공 시 `indexed` / 실패 시 `index_failed`.
- (선택) `report_status` 필드 추가: `none / generating / ready / failed` (별도 모델 미채택 시 필수).
- `error_pattern`: 리포트의 대표 오류유형을 채워 인덱싱 메타/집계에 활용.

### 5.3 choices 추가(안)
- `LLM_REQUEST_TYPE_CHOICES` += `("wrong_note_report", "오답리포트")`.
- 필요 시 `WRONG_NOTE_STATUS_CHOICES`에 `report_ready` 등 세분화(현 4값 유지도 가능).

## 6. RAG 문서 구조 1차 초안 (ChromaDB)

컬렉션 **`wrong_note_embeddings`** 유지. 재생성 시 `id = "wrong_note:{wrong_note_id}"` 로 **upsert**.

### 6.1 인덱싱 문서(document text)
증상(회고)과 진단(리포트)을 함께 담아 **양쪽 쿼리에 매칭**되게 한다.
```
문제명: {title}
난이도: {difficulty}
태그: {tag1, tag2, ...}
제출 결과: {result}   오류: {error_message}
사용자 회고: {comment}
[리포트] 놓친 부분: {missed_points ...}
[리포트] 오답 원인/핵심: {summary ...}
[리포트] 학습 방향: {learning_direction ...}
(선택) 제출 코드 요약: {code 앞부분/시그니처}
```
> 코드 전문은 노이즈가 크므로 **요약/시그니처만** 권장(원문은 RDB submission.code에 보존).

### 6.2 메타데이터(metadata)
```json
{
  "user_id": <int>,            // 검색 필터 필수
  "wrong_note_id": <int>,
  "problem_id": <int>,
  "problem_title": "<str>",
  "difficulty": "<str>",
  "tags": "array,loop,dp",     // 쉼표 문자열(Chroma list 비호환 회피)
  "error_pattern": "<str>",
  "source": "wrong_note",
  "has_report": true,
  "created_at": "<iso>",
  "indexed_at": "<iso>"
}
```

### 6.3 검색 파라미터 · 2쿼리
| 단계 | 쿼리 텍스트 | where | top_k | threshold |
|---|---|---|---|---|
| ① 1차(회고기반) | `user_comment(회고)` | `{user_id}` | 5 | 0.35 |
| ③ 최종(전체기반) | `회고 + missed_points + learning_direction`(enriched) | `{user_id}` | 5 | 0.35 |

- `score = 1 - cosine_distance`, `score ≥ threshold` 만 채택.
- 자기 자신(`wrong_note_id == 현재 노트`) 제외 필터 적용.

### 6.4 임베딩 백엔드
- 현재: **결정적 hash 임베딩(MVP)** — 외부 모델 불필요, 검증용. 의미 유사도 품질 낮음.
- 권장(다음 단계): 실제 임베딩 모델(예: OpenAI `text-embedding-3-small`)로 교체. 문서 구조/메타/필터는 **모델과 무관하게 그대로 재사용** 가능.

## 7. API/엔드포인트 초안

### 7.1 Django (RDB 소유 · 오케스트레이션)
| 화면/동작 | 메서드 | 경로 | 설명 |
|---|---|---|---|
| 오답노트 작성(저장만) | POST | `/wrongnotes/create/<submission_id>/` | AI 미호출로 변경(FR-1.3) |
| **AI 리포트 생성** | POST | `/wrongnotes/<note_id>/report/` | 에이전트 호출 → 리포트 저장 + 인덱싱 |
| 리포트 조회 | GET | `/wrongnotes/<note_id>/` | 최신 리포트 + stage2 유사노트 표시 |
| 복습 완료 | POST | `/wrongnotes/<note_id>/review/` | 현행 유지 |

### 7.2 FastAPI (Chroma + LLM 소유 · 스테이트리스 에이전트)
| 엔드포인트 | 설명 |
|---|---|
| **`POST /ai/wrong-note/report`** (신규) | 2단계 RAG 에이전트: ①검색 → ②LLM → ③검색. `{report, stage1_evidence, stage2_evidence}` 반환 |
| `POST /ai/wrong-note/embed` | 회고+리포트 문서 인덱싱(upsert) — 현행 재사용 |
| `POST /ai/wrong-note/search` | 단발 유사검색 — 현행 유지(리포트 내부에서 재사용) |
| `POST /ai/wrong-note/ask` | '내 노트에 물어보기' — 현행 유지 |

- 내부 호출은 `X-Internal-API-Key`/`X-Request-ID`(현행 `ai_proxy.call_fastapi`).
- 응답 status: `success / empty / failed`(근거 없음=empty, 리포트만 제시).

## 8. 청킹(Chunking) 전략

오답노트 1건을 ChromaDB에 어떻게 쪼개(또는 안 쪼개고) 임베딩할지의 정책. 검색 정밀도 ↔ 복잡도 트레이드오프.

### 8.1 옵션 비교
| 전략 | 단위 | 장점 | 단점 | 권장 |
|---|---|---|---|---|
| **A. 노트=1 청크** | 노트 전체 텍스트 1개 | 단순, 노트 단위 관리 쉬움, 재인덱싱 upsert 1건 | 텍스트 길면 벡터가 평균화돼 신호 희석 | **MVP 시작점** |
| **B. 필드 단위 멀티 청크** | 회고 / 원인·놓친부분 / 학습방향 / 코드요약 | 필드별 정밀 매칭(증상↔증상, 진단↔진단), 다양성 확보 | 청크 관리·그룹핑 필요, 인덱싱/삭제 복잡 | 노트가 길어지거나 정밀도 필요 시 |
| **C. 슬라이딩 윈도우** | 긴 코드/서술을 300~400토큰·overlap 10~15% | 매우 긴 문서 대응 | 코드 청킹은 노이즈 큼 | 코드 검색이 꼭 필요할 때만 |

### 8.2 권장 (단계적)
- **1단계(현행/MVP)**: **A** — 노트=1 청크. **코드는 전문 대신 요약/시그니처만** 포함(원문은 RDB `submission.code`에 보존, 벡터 노이즈 방지).
- **2단계**: **B(필드 단위)** 로 확장. 청크 규칙:
  - `id = "wrong_note:{id}#{chunk_type}"` (예: `wrong_note:12#retro`)
  - 공통 metadata에 **`wrong_note_id`(그룹키)**, `chunk_type`(retro/cause/direction/code), `chunk_index` 추가
  - 검색 후 **같은 `wrong_note_id`로 그룹핑**해 노트 단위로 환원(청크 중 최고 score 채택) → 한 노트가 결과에 중복 노출되지 않게
- 텍스트 청크 크기(실모델 전제): 300~500자, overlap 10~15%. 임베딩 모델 max token 대비 충분히 작게.

### 8.3 인덱싱 문서 청킹 예 (2단계)
```
retro     : "사용자 회고: {comment}"
cause     : "오답 원인/핵심: {summary.cause} / {summary.problem_core} · 놓친 부분: {missed_points}"
direction : "학습 방향: {learning_direction} · 태그: {tags}"
code(선택) : "제출 코드 요약: {code 시그니처/앞부분}"
```
> ①회고기반 검색은 `retro`·`cause` 청크에, ③enriched 검색은 `cause`·`direction` 청크에 잘 맞도록 필드를 배치.

## 9. 리트리빙(Retrieval) 전략

### 9.1 2-hop (증상 → 진단)
- **Stage1(회고기반)**: query = 사용자 회고(comment). 없으면 `title + tags` fallback(현 구현). → LLM 컨텍스트 확보용, 좁게.
- **Stage2(최종)**: query = `회고 + missed_points + learning_direction + summary.cause`(진단 반영 enriched). → 사용자에게 보여줄 최종 유사 오답노트.

### 9.2 하이브리드 필터 (vector + metadata)
- **필수**: `where={"user_id": <id>}` (본인 노트만, 격리).
- **선택 부스팅/필터**: 같은 `tags` 겹침 가점, `problem_id` 동일 반복 오답은 정책에 따라 포함/제외, `error_pattern` 동일 우선.
- **자기/중복 제외**: 현재 노트(`wrong_note_id`) 제외(구현됨). 필드 청킹 시 같은 `wrong_note_id` 청크 dedup(최고 score 1건).

### 9.3 파라미터 튜닝
- 현재 `top_k=5`, `threshold=0.35`. **hash 임베딩에선 0.35가 대부분 걸러냄**(예: 강한 쿼리 0.749, 약한 회고 쿼리 0.35 미만 → 0건). **실모델 전환 시 재튜닝**(0.25~0.40 A/B).
- 권장 패턴: 후보 과다추출(top_k=10~20) → 임계값 → (선택)재랭킹/MMR → 상위 3~5 반환.

### 9.4 다양성 · 재랭킹 (선택, 실모델 단계)
- **MMR**(λ≈0.5): 유사 노트가 한 유형에 몰리면 학습가치 저하 → 관련성-다양성 균형, 서로 다른 `error_pattern`/태그가 섞이게.
- **재랭킹**: 벡터 후보를 cross-encoder/LLM으로 재정렬(정밀도↑·비용↑). MVP 생략.

### 9.5 콜드스타트 · 빈 결과
- **인덱싱 시점**: 리포트 성공 후 인덱싱 → 리포트 없는 노트는 검색 대상 아님. 첫 노트들은 서로 안 잡히는 **콜드스타트** 존재. 필요 시 "작성 시 회고만으로 우선 인덱싱" 옵션 고려.
- **빈 결과**: stage 검색 0건이면 **리포트만 제시**(근거 빈 배열, status=success). 신규 사용자에서 흔함.

## 11. 오픈 이슈 / 다음 단계
- **동기 vs 비동기**: 리포트 생성이 LLM+2검색이라 수 초 소요 → 우선 동기 요청+프론트 로딩. 필요 시 잡 큐(ExecutionJob류)로 비동기화.
- **자기 제외/중복**: stage 검색에서 현재 노트 및 동일 submission 노트 제외 규칙 확정.
- **임베딩 모델 교체**(hash→실모델)와 재인덱싱 배치 필요.
- **회고 미작성 리포트**: 회고 없이 리포트 요청 시 코드/결과만으로 생성 허용할지 정책 결정.
- 관련 문서: 데이터모델 `6.` · AI/RAG/FE 연계 `7.` · TC 생성 에이전트 `8.`.
