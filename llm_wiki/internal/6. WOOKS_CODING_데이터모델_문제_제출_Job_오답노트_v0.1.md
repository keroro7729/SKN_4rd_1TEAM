작성일: 2026-07-03 · 기준: **코드생성 지시문 v0.7 §7 (PostgreSQL 데이터 모델)** + 스캐폴딩 실측
대상 STEP: **STEP-02 (데이터 모델)**. 이 문서는 STEP-02 구현의 설계 기준이다.

> ⚠️ 필드명·choices·인덱스는 지시문 §7/§7.1을 **그대로** 따른다(임의 변경 금지).

## 0. 핵심 결론 먼저

- **제출이력(Submission)과 실행작업(ExecutionJob)은 "분리"한다.** → 지시문 §7이 이미 별도 모델로 규정.
  근거 요약(§7): *"ExecutionJob.status는 처리 생명주기, Submission.result는 사용자에게 보일 최종
  채점 결과다. polling 응답은 둘 다 반환한다."* (상세 근거·비교는 §5)
- 문제 도메인은 **ProblemCategory · ProblemTag · Problem · TestCase** 4모델로 구성.
- 오답노트는 **WrongNote(원문/분석) + WrongNoteVector(임베딩 메타)** 2모델 + ChromaDB `wrong_note_embeddings`.

## 1. 코딩테스트 문제 모델

### 1.1 필드 (지시문 §7)

| 모델 | 필수 필드 | 제약/인덱스 |
|---|---|---|
| **ProblemCategory** | name, slug, order, is_active | slug unique |
| **ProblemTag** | name, slug | slug unique |
| **Problem** | category FK, tags M2M ProblemTag, title, description, difficulty, constraints, is_active | category/difficulty index |
| **TestCase** | problem FK, input_data, expected_output, is_sample, compare_mode, float_tolerance | problem_id index, compare_mode 기본 `line_trim` |

- `Problem.difficulty` ∈ `DIFFICULTY_CHOICES` = basic/beginner/intermediate/advanced (언어기초/초급/중급/고급)
- `TestCase.is_sample`: 사용자에게 공개되는 예제 케이스 여부(=샘플). 채점은 전체 케이스로.
- `TestCase.compare_mode` ∈ `TEST_COMPARE_MODE_CHOICES` = exact/line_trim/float
  - 기본 `line_trim`(줄단위 끝공백 정리, LF 정규화, **대소문자 구분**). float 문제만 `float`(=float_tolerance 사용).
  - 채점 로직은 `worker/runner.py:compare_output` — 현재 line_trim 구현, float은 STEP-04 확장.
- `tags`는 **Problem.tags(M2M ProblemTag)** 가 원천. 오답노트 draft 생성 시 이 태그를 WrongNote.tags로 복사(§10).

### 1.2 설계 스케치 (참고)

```python
class ProblemCategory(models.Model):
    name = models.CharField(max_length=50)
    slug = models.SlugField(unique=True)
    order = models.PositiveIntegerField(default=0)
    is_active = models.BooleanField(default=True)

class ProblemTag(models.Model):
    name = models.CharField(max_length=50)
    slug = models.SlugField(unique=True)

class Problem(models.Model):
    category = models.ForeignKey(ProblemCategory, on_delete=models.PROTECT, related_name="problems")
    tags = models.ManyToManyField(ProblemTag, blank=True, related_name="problems")
    title = models.CharField(max_length=200)
    description = models.TextField()
    difficulty = models.CharField(max_length=20, choices=DIFFICULTY_CHOICES, default="beginner")
    constraints = models.TextField(blank=True)          # 입력 제약/시간·메모리 안내
    is_active = models.BooleanField(default=True)
    created_at = models.DateTimeField(auto_now_add=True)
    class Meta:
        indexes = [models.Index(fields=["category", "difficulty"])]

class TestCase(models.Model):
    problem = models.ForeignKey(Problem, on_delete=models.CASCADE, related_name="test_cases")
    input_data = models.TextField(blank=True)
    expected_output = models.TextField(blank=True)
    is_sample = models.BooleanField(default=False)
    compare_mode = models.CharField(max_length=20, choices=TEST_COMPARE_MODE_CHOICES, default="line_trim")
    float_tolerance = models.FloatField(default=1e-6)
    class Meta:
        indexes = [models.Index(fields=["problem"])]
```

## 2. 제출(Submission) 모델 — 사용자에게 보이는 "최종 채점 결과"

| 모델 | 필수 필드 | 제약/인덱스 |
|---|---|---|
| **Submission** | user FK, problem FK, code, result, output, error_message, elapsed_ms | user/problem/created_at index |

- `result` ∈ `SUBMISSION_RESULT_CHOICES` = pending/success/wrong/error/timeout (대기/정답/오답/오류/시간초과)
- `output`: 사용자 코드 표준출력(대표 케이스), `error_message`: 사용자 코드 오류 메시지, `elapsed_ms`: 실행시간(ms)
- **역할**: "이 사용자가 이 문제를 이렇게 제출했고, 최종 채점은 이렇다"는 **도메인 기록(영구 보존)**.
  마이페이지/결과 화면은 Submission을 조회한다(§7 SubmissionResultView).
- 포인트 연동: 생성 시 `submission_created`(+10), `result=success` 시 `solve_success`(+20) 지급(중복 금지).

```python
class Submission(models.Model):
    user = models.ForeignKey(settings.AUTH_USER_MODEL, on_delete=models.CASCADE, related_name="submissions")
    problem = models.ForeignKey(Problem, on_delete=models.CASCADE, related_name="submissions")
    code = models.TextField()
    result = models.CharField(max_length=20, choices=SUBMISSION_RESULT_CHOICES, default="pending")
    output = models.TextField(blank=True)
    error_message = models.TextField(blank=True)
    elapsed_ms = models.PositiveIntegerField(null=True, blank=True)
    created_at = models.DateTimeField(auto_now_add=True)
    class Meta:
        indexes = [models.Index(fields=["user", "problem", "created_at"])]
```

## 3. 실행작업(ExecutionJob) 모델 — Worker가 처리하는 "운영 단위"

| 모델 | 필수 필드 | 제약/인덱스 |
|---|---|---|
| **ExecutionJob** | user FK, submission FK, job_type, status, input_payload, result_payload, retry_count, max_retry, worker_id, started_at, finished_at | status/created_at index |

- `job_type` ∈ `JOB_TYPE_CHOICES` = **`code_run`** (코드실행)  ⚠️ 아래 §7 불일치 주의
- `status` ∈ `JOB_STATUS_CHOICES` = pending/running/success/failed/timeout
- `input_payload`/`result_payload`: **JSONField**. 실행 입력(코드·stdin 등)·결과(stdout/stderr/returncode 등) 원본.
- `retry_count`/`max_retry`: Worker 장애 시 재시도 제어. `worker_id`: 어느 워커가 잡았는지.
- **역할**: Redis/Celery 없이 **PostgreSQL polling 큐**로 동작(§9). Worker가 `FOR UPDATE SKIP LOCKED`로
  pending 1건 선점 → running → 실행 → success/failed/timeout + Submission.result 갱신.

```python
class ExecutionJob(models.Model):
    user = models.ForeignKey(settings.AUTH_USER_MODEL, on_delete=models.CASCADE, related_name="jobs")
    submission = models.ForeignKey(Submission, on_delete=models.CASCADE, related_name="jobs")
    job_type = models.CharField(max_length=20, choices=JOB_TYPE_CHOICES, default="code_run")
    status = models.CharField(max_length=20, choices=JOB_STATUS_CHOICES, default="pending")
    input_payload = models.JSONField(default=dict)
    result_payload = models.JSONField(default=dict)
    retry_count = models.PositiveIntegerField(default=0)
    max_retry = models.PositiveIntegerField(default=1)
    worker_id = models.CharField(max_length=64, blank=True)
    started_at = models.DateTimeField(null=True, blank=True)
    finished_at = models.DateTimeField(null=True, blank=True)
    created_at = models.DateTimeField(auto_now_add=True)
    class Meta:
        indexes = [models.Index(fields=["status", "created_at"])]
```

## 4. 오답노트 모델

| 모델 | 필수 필드 | 제약/인덱스 |
|---|---|---|
| **WrongNote** | user FK, problem FK, submission FK, tags M2M ProblemTag, comment, ai_analysis, error_pattern, status, is_reviewed, reviewed_at | user/status/created_at index |
| **WrongNoteVector** | user FK, wrong_note FK, embedding_id, source, indexed_at | wrong_note unique, user_id index |
| **WrongNoteQueryLog** | user FK, query, answer, evidence_note_ids, scores | user/created_at index |

- `status` ∈ `WRONG_NOTE_STATUS_CHOICES` = draft/completed/indexed/index_failed (작성중/완료/인덱싱완료/인덱싱실패)
  - 작성 화면 진입 시 `submission_id` 기준 **draft를 먼저 생성**해 `wrong_note_id` 확보(§10).
  - `completed` 전환 시 포인트 `wrongnote_completed`(+15). 이후 ChromaDB 반영되면 `indexed`.
- `ai_analysis`: **JSONField** — FastAPI `/ai/wrong-note/analyze` 4섹션(problem_core/cause/solution/caution) 저장.
- `error_pattern`: 사용자 코멘트에서 보정한 오류 유형(태그와 함께 RAG 품질에 사용).
- **WrongNoteVector**: PostgreSQL 저장 후 ChromaDB `wrong_note_embeddings` 반영 결과의 **메타 기록**
  (embedding_id, source, indexed_at). 실제 벡터는 ChromaDB, 원문은 WrongNote. **1:1(unique)**.

```python
class WrongNote(models.Model):
    user = models.ForeignKey(settings.AUTH_USER_MODEL, on_delete=models.CASCADE, related_name="wrong_notes")
    problem = models.ForeignKey(Problem, on_delete=models.CASCADE, related_name="wrong_notes")
    submission = models.ForeignKey(Submission, on_delete=models.CASCADE, related_name="wrong_notes")
    tags = models.ManyToManyField(ProblemTag, blank=True, related_name="wrong_notes")
    comment = models.TextField(blank=True)
    ai_analysis = models.JSONField(default=dict, blank=True)
    error_pattern = models.CharField(max_length=100, blank=True)
    status = models.CharField(max_length=20, choices=WRONG_NOTE_STATUS_CHOICES, default="draft")
    is_reviewed = models.BooleanField(default=False)
    reviewed_at = models.DateTimeField(null=True, blank=True)
    created_at = models.DateTimeField(auto_now_add=True)
    class Meta:
        indexes = [models.Index(fields=["user", "status", "created_at"])]

class WrongNoteVector(models.Model):
    user = models.ForeignKey(settings.AUTH_USER_MODEL, on_delete=models.CASCADE)
    wrong_note = models.OneToOneField(WrongNote, on_delete=models.CASCADE, related_name="vector")
    embedding_id = models.CharField(max_length=128)
    source = models.CharField(max_length=30, default="wrong_note")
    indexed_at = models.DateTimeField(null=True, blank=True)
    class Meta:
        indexes = [models.Index(fields=["user"])]
```

> ChromaDB `wrong_note_embeddings` metadata: `user_id, note_id, problem_id, tags, source, created_at`.
> tags는 list 저장 호환성 때문에 **쉼표 문자열**(`"array,loop,condition"`)로 저장(§10).

## 5. ★ 제출이력: Job에 포함 vs 별도 모델 — 결정과 근거

### 결정: **별도 모델(Submission ≠ ExecutionJob)로 분리한다.**
지시문 §7이 이미 두 모델을 별도로 규정하며, 그 이유를 명시한다.

### 왜 분리가 더 좋은가 (관심사 분리)

| 관점 | Submission (제출이력) | ExecutionJob (실행작업) |
|---|---|---|
| 성격 | **도메인 기록** (사용자가 보는 최종 채점) | **운영/처리 단위** (Worker 큐잉) |
| 보존 | 영구 (이력·통계·마이페이지) | 처리 후에도 남지만 관리자/운영용 |
| 상태 | result: pending→success/wrong/error/timeout (**1회 확정**) | status: pending→running→success/failed/timeout (**재시도 가능**) |
| 고유 필드 | code, output, error_message, elapsed_ms | retry_count, max_retry, worker_id, started/finished_at, payload |
| 조회 주체 | 사용자 화면(SubmissionResultView) | 관리자/운영(ErrorLog·LLMRequestLog·Job 상태 조회) |
| 카디널리티 | 제출 1건 | 그 제출에 대한 실행 **1..N**(재시도 시 증가) |

- **1 Submission ─ N ExecutionJob**: Job은 `submission FK`를 가진다(재시도/재큐잉 시 Job이 늘어도 제출이력은 하나로 유지).
- **polling 응답은 둘 다 반환**: `job_status`(처리 진행) + `submission_result`(최종 채점) → 화면은 job이 끝나면
  submission_result로 확정 표시(§9.4 매핑 참조).

### 만약 하나로 합쳤다면 (안티패턴)
- 재시도 메타데이터(worker_id/retry)가 사용자 이력 레코드를 오염.
- 재시도할 때마다 같은 행을 덮어써 **제출 이력이 소실**되거나, 운영 필드가 이력 통계 쿼리에 섞임.
- "처리 생명주기"와 "최종 결과"가 한 컬럼에 뭉쳐 상태 전이가 모호해짐.

→ **분리 유지.** (합치는 최적화는 이 서비스 규모/요구에서 이득 없음.)

## 6. 관계(ERD) 요약

```
CustomUser 1─┬─N Submission ─1─N ExecutionJob        (Job.submission FK)
             ├─N WrongNote                            (제출당 오답노트 0..1)
             └─N ExecutionJob / WrongNoteVector ...

ProblemCategory 1─N Problem
ProblemTag      N─M Problem,  N─M WrongNote
Problem 1─┬─N TestCase
          ├─N Submission
          └─N WrongNote

Submission 1─0..1 WrongNote (submission FK) ─ 1:1 ─ WrongNoteVector (wrong_note unique)
                                                      └→ ChromaDB wrong_note_embeddings
```

핵심 흐름(§9):
```
Browser → Django /submissions/run/ → Submission+ExecutionJob(pending) 생성
        → Worker polling(선점) → run_code/compare_output
        → Submission.result + ExecutionJob.status 저장 → 결과 화면
오답노트: submission 기준 WrongNote(draft) → FastAPI search/analyze → completed → embed → indexed
```

## 7. ⚠️ 스캐폴딩과의 불일치 (STEP-02/04에서 정합)

- **`job_type` 값 불일치**: 스펙 §7.1 = **`code_run`** 이지만, 현재 `worker/main.py` SQL은
  `job_type='code_execution'` 로 되어 있다. → **모델은 스펙대로 `code_run`**, Worker SQL을
  STEP-04에서 `code_run` 으로 정합. (지시문 "상태값 변경 금지" 원칙상 **스펙 값이 정답**.)
- **모델명**: 벡터 메타 모델은 스펙상 **`WrongNoteVector`** (FastAPI 임베딩 엔드포인트 이름 `embed`와 혼동 주의).
- Worker `jobs` polling SQL의 테이블/컬럼명은 STEP-02 실제 모델(`execution_job`, `submission`)에 맞춰 확정.
