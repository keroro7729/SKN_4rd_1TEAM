## 1. 저장소(볼륨)와 데이터 영속성

`docker-compose.yml` 기준 상태 저장소는 **Docker named volume** 2개다.

| 서비스 | 컨테이너 경로 | named volume | 재시작 시 |
|---|---|---|---|
| postgres | `/var/lib/postgresql/data` | `pgdata` | 유지 ✅ |
| chromadb | `/chroma/chroma` | `chromadata` | 유지 ✅ |
| (django collectstatic) | `/app/staticfiles` | `staticfiles` | 유지 |
| **logs (전 컴포넌트)** | `/app/logs` | **bind mount `./logs`** | 유지(호스트 파일) |

- **저장 위치**: named volume 은 호스트 리포 폴더가 아니라 Docker 관리 스토리지에 있다
  (Windows/WSL2 는 Docker Desktop VM 내부). `docker volume inspect pgdata` 로 확인.
- **영속성 규칙**:
  - `restart` / `down` → `up` / `up --build` : **데이터 유지**
  - `docker compose down -v` : **volume 삭제 → 데이터 소멸** (이때만 날아감)
- **로그는 예외적으로 bind mount** (`./logs:/app/logs`) → 컨테이너를 지워도 호스트
  `logs/` 에 그대로 남아 바로 열람/분석 가능. (로깅 설계는 문서 5 참고)

## 2. 현재 실측 접근 관계 (중요)

"Django·FastAPI 둘 다 RDB·VDB 를 접근"하는 구조가 **아니다.** 실제 코드 기준:

| 컴포넌트 | PostgreSQL(RDB) | ChromaDB(VDB) | 근거 |
|---|:---:|:---:|---|
| **Django** | ✅ | ❌ | `config/settings.py` DATABASES. Chroma 의존성 없음 |
| **FastAPI** | ❌ | ✅ | psycopg 미포함. 데이터는 요청 body 로 수신 |
| **Worker** | ✅ | ❌ | `jobs` 테이블 polling (psycopg) |

- **FastAPI 는 RDB 에 직접 붙지 않는 stateless AI 연산 서버**다. 필요한 값
  (`problem_id`, `user_code`, `user_id`, `content` 등)을 전부 Django 요청 body 로 받는다.
- **RDB 소유/공유**: Django(쓰기 주인) + Worker(실행 상태 갱신)가 공유.
- **VDB 소유**: FastAPI 단독.

### "벡터DB는 FastAPI만 접근 가능한가?"
- **설계상 예**(역할 분리). 오답노트 인덱싱도 `Django(RDB 저장) → HTTP → FastAPI(/ai/wrong-note/embed) → Chroma` 경로로만 이뤄진다.
- **네트워크상 아니오**. compose 기본 bridge 네트워크에서는 서비스명으로 서로 접근 가능하고
  Chroma 에 인증이 없어, 원하면 Worker/Django 도 `http://chromadb:8000` 에 붙을 수 있다.
  → **"FastAPI만 접근"은 네트워크가 강제하는 게 아니라 우리가 지키는 규율(convention)**이다.

### 외부 노출 현황
| 서비스 | host 포트 | 비고 |
|---|---|---|
| nginx | `80:80` | ✅ 유일한 정식 외부 관문 |
| **postgres** | **`5432:5432`** | ⚠️ host 에 열림 — 로컬 개발엔 편하나 **EC2 배포 시 닫을 것** |
| chromadb | 없음 | 내부 네트워크 전용(host 접근 불가) ✅ |
| django/fastapi/worker | 없음 | 내부 전용 |

## 3. 의존구조 규칙 (SSOT — Single Source of Truth)

### 3.1 데이터 소유권 (하나씩만)
```
PostgreSQL  ← 쓰기 주인: Django   (Worker 는 jobs/submissions 상태만 갱신하는 실행자)
ChromaDB    ← 쓰기 주인: FastAPI  (Django 는 절대 직접 접근하지 않음)
```
- ❌ **FastAPI 에 psycopg 추가 금지** — RDB 직접 접근은 소유권을 깨뜨린다. payload 로만 수수.
- ❌ **Django 에 chromadb 클라이언트 추가 금지** — 벡터 작업은 전부 FastAPI HTTP 경유.

### 3.2 호출 방향 (단방향, 순환 금지)
```
사용자 → Nginx → Django ─(X-Internal-API-Key)→ FastAPI → (Chroma / OpenAI)
                    │
                    └→ PostgreSQL ←(polling)─ Worker
```
- FastAPI → Django **역호출 금지**(순환 의존 방지). FastAPI 는 받은 요청의 응답만 반환.
- FastAPI 내부 API 는 `Depends(verify_internal)` 로 Django 호출만 허용(현재 반영됨).

### 3.3 앞으로의 강화 항목(개선 백로그)
- **네트워크 분리**: compose 에 `backend`/`ai` 네트워크를 나눠 Worker/Django 가
  chromadb 와 같은 네트워크에 없도록 하면 물리적으로 차단(규율 → 강제).
- **postgres host 포트 제거**: 배포 시 `ports: 5432:5432` 삭제(내부 전용화).
- **Chroma 인증**: `CHROMA_SERVER_AUTHN_*` 토큰 도입.
- **`depends_on` 준비성**: 현재 `depends_on` 은 "시작 순서"만 보장하고 "준비 완료(ready)"는
  보장하지 않는다. postgres 미준비 상태에서 Django `migrate` 가 먼저 돌면 첫 기동 실패 가능 →
  `healthcheck` + `depends_on: { condition: service_healthy }` 로 견고화(STEP-09).

## 4. 요약
- 재시작에도 **데이터 유지**(오직 `down -v` 만 삭제). 로그는 bind mount 로 호스트에 상시 보존.
- **FastAPI=VDB 전용·stateless / Django=RDB 전용 / Worker=RDB 실행자**가 현재 실제 구조.
- VDB 격리는 **규율**일 뿐 네트워크가 막지 않음 → 소유권 단일화 + 단방향 호출 +
  (다음 단계) 네트워크 분리·healthcheck 로 굳힌다.
