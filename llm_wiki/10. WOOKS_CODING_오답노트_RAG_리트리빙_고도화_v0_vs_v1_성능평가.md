# 10. 오답노트 RAG 리트리빙 고도화 — v0 vs v1 성능평가

> 설계 근거: 문서 9 §8-9(청킹/리트리빙 전략)의 **2단계(필드 단위 멀티 청크)** 를 구현하고,
> 사용자 요구(섹션별 유사도 mean·max 종합 스코어링)를 반영한 개선안(v1)의 실측 평가.

## 1. 목표

유사 오답노트 검색 정확도를 올린다. 핵심 가설:

1. **노이즈 제외** — 문제 원문·제출 코드·난이도·에러·결과처럼 유형 매칭을 방해하는 정보를 인덱스에서 제외한다. 문제 **카테고리·알고리즘 분류** 만 가볍게 추가 고려한다.
2. **섹션 분리 청킹** — 회고 + AI 생성 코멘트를 **짧은 청크로 분리 임베딩**(노트당 멀티 벡터)해 정밀 매칭한다.
3. **mean/max 종합 스코어링** — 후보 노트를 **섹션별 유사도**로 환원한 뒤 평균(mean)·최댓값(max)을 가중 종합해 랭킹한다.

## 2. v0 vs v1 설계 비교

| 항목 | v0 (기존) | v1 (개선) |
|---|---|---|
| 인덱싱 단위 | 노트 **1개 = 1벡터** | 노트 **1개 = 섹션 N청크(멀티 벡터)** |
| 인덱스 포함 텍스트 | 문제명·난이도·태그·회고·결과·에러·**코드**·분석 전부 | **회고 + AI 코멘트(문제핵심/풀이/원인/개선/피드백/체크)** 만 |
| 제외(노이즈) | — | **문제 원문·코드·난이도·에러·결과** |
| 추가 고려 | (블롭에 섞임) | **카테고리·알고리즘 분류** = `topic` 청크 1개 |
| 검색 쿼리 | `문제명 + 회고 + 태그` 1벡터 | **회고 청크들 + topic** 다벡터 |
| 스코어링 | 쿼리-블롭 **단일 cosine** | 후보 **섹션별 max sim → mean·max 가중 종합** |
| 자기/중복 | note_id로 제외 | note_id로 제외 + 청크는 노트 단위로 환원 |

### 2.1 스코어링 공식 (v1)

후보 노트 N에 대해, 각 섹션 s의 유사도를 “그 섹션의 어떤 청크가 어떤 쿼리 청크와 가장 잘 맞는가”로 정의:

```
section_sim[s] = max over (query chunk q, candidate chunk c ∈ s) cosine(q, c)
final(N)       = ( W_MEAN · mean_s(section_sim) + W_MAX · max_s(section_sim) ) / (W_MEAN + W_MAX)
```

- `mean` : 여러 섹션에서 고르게 닮은 노트를 우대(전반적 유사).
- `max` : 한 섹션이라도 강하게 일치하는 노트를 우대(핵심 증상 일치).
- 기본 가중 `W_MEAN = W_MAX = 0.5`. `RAG_SCORE_THRESHOLD` 미만은 컷.

## 3. 구현 위치

| 계층 | 파일 | 변경 |
|---|---|---|
| FastAPI RAG | `fastapi_app/services/chroma.py` | 청킹(`chunk_text`)·청크 구성(`build_note_chunks`)·집계(`aggregate_section_sims`)·인덱싱(`embed_note_chunked`)·검색(`search_notes`) |
| FastAPI 임베딩 | `fastapi_app/services/chroma.py` | `embed_text` = **OpenAI 임베딩(`text-embedding-3-small`)** + 해시 폴백, 백엔드별 **컬렉션 버저닝**(`_collection_name`) |
| FastAPI 라우터 | `fastapi_app/routers/wrong_note.py` | `/embed`(섹션), `/search`(회고+topic 쿼리) |
| FastAPI 스키마 | `fastapi_app/schemas/wrong_note.py` | `WrongNoteEmbedRequest(sections/category/algo_tags)`, `WrongNoteSearchRequest(category/wrong_note_id)` |
| Django | `django/wrongnotes/services.py` | `build_wrong_note_sections`(노이즈 제외·섹션 구성), embed payload, 검색 payload에 `category` |
| 파라미터 | `fastapi_app/config.py` | `RAG_CHUNK_MAX_CHARS=160`, `RAG_PER_QUERY_K=12`, `RAG_W_MEAN=0.5`, `RAG_W_MAX=0.5`, `EMBED_MODEL`, `RAG_EMBED_BACKEND` |
| 평가 | `fastapi_app/services/rag_eval.py` | 합성 벤치마크 + v0/v1 파이프라인 + 지표 |

> 데이터 소유권 유지: 청킹·임베딩·검색은 전부 **FastAPI(ChromaDB 소유자)** 도메인. Django는 섹션 텍스트만 payload로 전달(문제 원문/코드는 인덱스에 넣지 않음).

## 4. 평가 방법론

**합성 벤치마크(라벨 有)** — `services/rag_eval.py`. ChromaDB 없이 순수 함수로 재현.

- **4개 오류 유형 클러스터 × 5노트 = 20 오답노트.**
  - 클러스터: `배열 인덱스 범위 초과` / `정수 오버플로우` / `재귀 종료조건 누락` / `부동소수점 비교 오차`.
  - **신호**: 회고·AI 코멘트를 유형별로 구분(클러스터 내부는 패러프레이즈로 어휘 공유).
  - **노이즈**: 문제 원문·코드를 **유형과 무관한** 5개 문제 주제(정렬/그래프/문자열/약수/계단)로 배정 → 같은 주제가 여러 클러스터에 걸쳐 등장(실제 코테: 같은 실수도 문제마다 지문·코드가 다름).
- **정답(relevant)** = 같은 클러스터의 다른 노트(쿼리당 4개).
- **파이프라인은 실제 코드 경로를 반영**:
  - v0: index=단일 블롭, query=`문제명+회고+태그`, score=cosine.
  - v1: index=섹션 청크, query=`회고 청크+topic`(분석 전 상태 재현), score=섹션 mean/max 집계.
- **지표(k=4, 20쿼리 평균)**: Precision@4, Recall@4, MRR, nDCG@4.

## 5. 결과

### 5.1 실측 (임베딩 = OpenAI `text-embedding-3-small`)

| metric | v0 (단일 블롭) | v1 (섹션 청킹) | 개선율 |
|---|---:|---:|---:|
| Precision@4 | 0.6250 | **1.0000** | **+60.0%** |
| Recall@4 | 0.6250 | **1.0000** | **+60.0%** |
| MRR | 0.7417 | **1.0000** | **+34.8%** |
| nDCG@4 | 0.6182 | **1.0000** | **+61.8%** |

**정성 예시** — 쿼리 노트 #1(클러스터 `배열 인덱스 범위 초과`) 상위 5:

```
v0: #11(X,0.51), #6(X,0.51), #16(X,0.50), #4(O,0.45), #3(O,0.42)
v1: #3 (O,0.77), #5(O,0.76), #4(O,0.76), #2(O,0.75), #7(X,0.57)
```

- **v0는 상위 3건이 전부 오답**(#11·#6·#16). 이들은 #1과 **문제 주제(코드/지문)가 같은** 교차-클러스터 노트다 → v0가 노이즈에 낚였다.
- **v1은 상위 4건이 전부 정답**(같은 오류 유형) → 회고·AI 코멘트 신호를 청킹으로 분리해 노이즈를 배제.

라이브 ChromaDB 왕복(실제 인덱싱→검색)에서도 동일 경향 확인(top-4 정답, 컬렉션 `wrong_note_embeddings-oa-text-embedding-3-small`).

### 5.2 임베더 영향 분석 (중요한 실측 발견)

동일한 v1 구조라도 **임베더에 따라 결과가 정반대**였다.

| 임베딩 백엔드 | v1 Precision@4 | 비고 |
|---|---:|---|
| 결정적 해시 BoW (64→256차원) | 0.24 ~ 0.54 (**v0보다 낮음**) | 짧은 한국어 청크에서 **조사·공통어**가 유사도 하한을 높여 변별력 붕괴, 저차원 해시 충돌 |
| OpenAI `text-embedding-3-small` | **1.00** (v0=0.625) | 시맨틱 매칭 → 짧은 청크가 정밀도로 전환 |

**결론**: “짧은 청크 분리 매칭”은 **시맨틱 임베딩을 전제**한다. 토큰-해시 BoW는 짧은 청크에서 어휘 겹침(특히 한국어 조사)이 지배해 오히려 신호를 희석한다. 따라서 v1의 라이브 기본 백엔드를 **OpenAI 임베딩**으로 두고, 키가 없을 때만 결정적 해시로 폴백한다(`RAG_EMBED_BACKEND=auto`).

> 재현: `docker compose run --rm fastapi python -m services.rag_eval`
> (해시 백엔드로 재현하려면 `RAG_EMBED_BACKEND=hash`.)

## 6. 운영 / 마이그레이션 노트

- **컬렉션 버저닝**: 임베더가 바뀌면 벡터 차원이 달라지므로 컬렉션명을 `wrong_note_embeddings-<임베더서명>` 으로 분리(`_collection_name`). 기존 v0 컬렉션은 **삭제하지 않고** 방치(비파괴). 실행/차원 충돌 방지.
- **기존 노트 재인덱싱 필요**: v0로 인덱싱된 노트는 v1 구조(섹션 청크)로 다시 임베딩해야 개선 효과가 난다. 오답노트 저장/재분석 시 `embed_wrong_note` 가 `embed_note_chunked` 로 재인덱싱하며, 재인덱싱 시 해당 노트의 이전 청크를 `delete(where={wrong_note_id})` 후 재생성한다.
- **파라미터 튜닝 지점**(`.env`): `RAG_CHUNK_MAX_CHARS`(청크 길이), `RAG_W_MEAN`/`RAG_W_MAX`(집계 가중), `RAG_PER_QUERY_K`(쿼리 청크당 후보 수), `RAG_SCORE_THRESHOLD`, `EMBED_MODEL`.
- **비용**: 임베딩은 노트당 청크 수만큼 호출(약 6~8회). `text-embedding-3-small` 기준 미미. 검색은 쿼리 청크 수만큼 질의.

## 7. 한계와 향후 과제

- 벤치마크는 **합성 데이터**(20노트)다. 절대 수치(v1=1.00)는 데이터가 잘 분리되게 구성된 상한선에 가깝다 — 실제 노트로 재측정 필요.
- 섹션 정렬은 **후보 섹션 기준 max**(쿼리는 회고만 있는 검색 시점 상황). 회고↔원인 같은 **교차-섹션 정렬**을 명시적으로 가중하면 더 정밀할 수 있음.
- 실서비스에서 노트 수가 커지면 `RAG_PER_QUERY_K`·후보 컷·재랭킹 비용 재점검 필요(현재는 user 스코프라 규모 제한적).
- 문서 9의 **2단계 리트리빙(회고 1차 → enriched 최종)** 과 결합 시 stage별 청크 타입 가중 실험 여지.
