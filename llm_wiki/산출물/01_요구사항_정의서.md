## 1. 서비스 정의

WOOK'S CODING은 Python 코딩 문제 풀이, 격리 코드 실행·채점, AI 단계별 힌트, 오답노트 분석 및 개인별 RAG 검색을 통합 제공하는 웹 기반 학습 서비스다.

## 2. 범위

### 2.1 포함 범위

- 회원가입, 로그인, 로그아웃, 계정 관리 및 권한 관리
- 문제 조회·검색·필터·즐겨찾기·추천
- Python 코드 실행, 최종 제출, 비동기 채점 및 결과 조회
- 테스트케이스 자동 생성
- AI 단계별 힌트, 미니튜터, 사용자 코딩 상태 요약
- 오답노트 작성·AI 분석·벡터 인덱싱·검색·복습
- 포인트, 레벨, 아바타, 미션
- 공지사항, 마이페이지, 관리자 대시보드, AI 실험실, 로그 조회
- Docker Compose 기반 로컬 및 AWS 운영 배포

### 2.2 제외 범위

- 결제 및 유료 구독 처리
- Python 이외 언어의 실제 실행·채점
- FastAPI, Worker, PostgreSQL, ChromaDB의 외부 직접 공개
- 네이티브 모바일 애플리케이션

### 4.4 코드 실행·채점

| ID | 요구사항 | 수용 기준 |
|---|---|---|
| FR-EXEC-001 | 사용자는 풀이 코드를 실행할 수 있어야 한다. | 실행 요청은 `run` 제출과 `code_run` 작업을 생성하고 빠른 테스트케이스 1건으로 검증한다. |
| FR-EXEC-002 | 사용자는 코드를 최종 제출할 수 있어야 한다. | 최종 제출은 `submit` 제출과 `code_submit` 작업을 생성하고 전체 테스트케이스를 검증한다. |
| FR-EXEC-003 | 실행·제출은 비동기로 처리되어야 한다. | 최초 응답은 HTTP 202이며 클라이언트는 결과 API를 폴링해 `generating/pending/running/success/failed/timeout` 상태를 확인한다. |
| FR-EXEC-004 | 테스트케이스가 없는 문제는 최초 실행 시 자동 생성되어야 한다. | 문제 단위 잠금과 생성 마커로 중복 생성을 방지하고, 생성 완료 후 실행 작업이 등록된다. |
| FR-EXEC-005 | 채점은 `exact`, `line_trim`, `float` 비교 방식을 지원해야 한다. | 설정된 비교 방식과 부동소수 허용 오차에 따라 정오가 판정된다. |
| FR-EXEC-006 | 결과는 정답·오답·오류·시간초과를 구분해야 한다. | 출력, 오류 메시지, 실행 시간, 통과·전체 테스트 수와 케이스별 결과가 반환된다. |
| FR-EXEC-007 | 실행 작업이 장시간 미완료 상태로 남지 않아야 한다. | 설정된 결과 제한시간을 넘으면 실패로 종료하고 Worker 점검 안내를 반환한다. |
| FR-EXEC-008 | 실패한 최종 제출은 오답노트 작성으로 연결되어야 한다. | `wrong/error/timeout`인 최종 제출에만 오답노트 작성 URL이 반환된다. |

## 6. 인터페이스 요구사항

`/`, `/accounts/`, `/problems/`, `/submissions/`, `/wrongnotes/`, `/mypage/`, `/notices/`, `/adminpanel/`, `/admin/`, `/health/`

### 6.2 Django 비동기·AI 프록시

- 제출: `POST /submissions/run/`, `POST /submissions/submit/`, `GET /submissions/{id}/result/`
- AI: `/ai-proxy/hint/`, `/ai-proxy/wrong-note/search/`, `/analyze/`, `/embed/`, `/ai-proxy/tutor/ask/`
- 상태 변경 요청은 로그인, HTTP 메서드 제한, CSRF 보호를 적용한다.

### 6.3 내부 FastAPI

- `/ai/health`, `/ai/hint`, `/ai/wrong-note/*`, `/ai/tutor/chat`, `/ai/coding-state/summarize`, `/ai/authoring/*`, `/ai/lab/*`, `/ai/diagnostics/chroma`
- 헬스체크를 제외한 내부 API는 `X-Internal-API-Key`와 요청 추적 ID를 검증한다.
- FastAPI는 외부에 직접 노출하지 않고 Django만 호출한다.

## 7. 비기능 요구사항

| ID | 구분 | 요구사항 |
|---|---|---|
| NFR-SEC-001 | 인증·인가 | 개인 데이터와 관리자 기능은 인증·소유권·역할 검사를 적용한다. |
| NFR-SEC-002 | 비밀정보 | `.env`는 버전 관리에서 제외하고 운영 키·비밀번호는 강한 값으로 교체한다. |
| NFR-SEC-003 | 네트워크 | 운영 외부 공개는 Nginx 80/443만 허용하고 DB·FastAPI·Worker·ChromaDB는 내부망에 둔다. |
| NFR-SEC-004 | 코드 격리 | 사용자 코드는 별도 Worker에서 실행하며 시간, CPU, 메모리, 파일 쓰기, 네트워크를 제한한다. |
| NFR-REL-001 | 작업 신뢰성 | Worker는 `FOR UPDATE SKIP LOCKED`로 작업을 선점하고 실패·시간초과를 종결 상태로 기록한다. |
| NFR-PERF-001 | 응답성 | 코드 실행과 AI·테스트케이스 생성의 장기 작업은 웹 요청을 차단하지 않아야 한다. |
| NFR-OBS-001 | 관측성 | Django, FastAPI, Worker 로그와 DB 오류·LLM 요청 로그를 보존한다. |
| NFR-DATA-001 | 정합성 | 상태 코드와 선택 값은 Django·FastAPI·Worker 간 동일하게 유지한다. |
| NFR-DEP-001 | 로컬 배포 | `docker-compose.yml`로 PostgreSQL, ChromaDB, Django, FastAPI, Worker, Nginx를 일괄 실행한다. |
| NFR-DEP-002 | 운영 배포 | 앱 EC2는 `compose/main.yml`, Worker EC2는 `compose/worker.yml`, 공용 DB는 RDS PostgreSQL을 사용한다. |
| NFR-MAINT-001 | 유지관리 | 마이그레이션, 정적 파일 수집, 헬스체크와 환경변수 기반 설정을 지원한다. |

## 8. 배포 구성

| 환경 | 구성 |
|---|---|
| 로컬·통합검증 | PostgreSQL + ChromaDB + Django/Gunicorn + FastAPI + Worker + Nginx |
| 운영 앱 서버 | Django/Gunicorn + FastAPI + ChromaDB + Nginx |
| 운영 Worker 서버 | Worker 단독 |
| 운영 공용 데이터베이스 | AWS RDS PostgreSQL |

운영 환경은 `DJANGO_DEBUG=False`, 운영용 `DJANGO_SECRET_KEY`, `DJANGO_ALLOWED_HOSTS`, `INTERNAL_API_KEY`, `OPENAI_API_KEY`, RDS 접속정보를 필수 설정한다.

## 9. 최종 인수 기준

1. 전체 Django 자동 테스트가 성공한다.
2. 로컬·앱·Worker Compose 설정 검증이 성공한다.
3. 회원가입부터 문제 풀이, 최종 제출, Worker 채점, 오답노트 작성·분석·인덱싱·복습까지의 핵심 흐름이 성공한다.
4. 타 사용자 데이터와 관리자 화면에 대한 비인가 접근이 차단된다.
5. 정답, 오답, 런타임 오류, 시간초과가 각각 올바른 상태와 메시지로 종료된다.
6. FastAPI와 ChromaDB는 외부에서 직접 접근할 수 없고 내부 인증이 적용된다.
7. 운영 앱 서버와 Worker 서버가 동일 RDS를 사용하며 앱 서버에는 Worker·PostgreSQL이 실행되지 않는다.
8. 운영 배포 자동화가 `main`, `compose/main.yml`, `compose/worker.yml` 기준으로 동작한다.
