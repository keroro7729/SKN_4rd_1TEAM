# 부록 A. 검증 절차 및 시나리오

> 저장소 clone 후 재현 가능한 검증 절차와 시나리오. 기준 브랜치 `main`. 운영은 앱 EC2(`compose/main.yml`) / 워커 EC2(`compose/worker.yml`) / 공용 RDS(PostgreSQL).

## A.1 사전 준비
```bash
git clone <repo> && cd <repo> && git switch main
cp .env.example .env   # DJANGO_SECRET_KEY·POSTGRES_*·INTERNAL_API_KEY·OPENAI_API_KEY 등 채움
```
- `.env`는 Git 미추적, 예시 비밀값 금지. AI 품질 검증 시 실제 `OPENAI_API_KEY` 설정.

## A.2 정적·구성 검증 (종료코드 0)
```bash
python -m compileall -q django fastapi_app worker
docker compose --env-file .env -f docker-compose.yml config --quiet
docker compose --env-file .env -f compose/main.yml   --project-directory . config --quiet
docker compose --env-file .env -f compose/worker.yml --project-directory . config --quiet
```
- 앱 서버 서비스 = `chromadb·django·fastapi·nginx`, 워커 서버 서비스 = `worker`만.

## A.3 로컬 통합 기동 · 자동 테스트
```bash
docker compose --env-file .env up --build -d           # postgres·chromadb·django·fastapi·worker·nginx
docker compose exec -T django python manage.py migrate --check
docker compose exec -T django python manage.py test    # 자동 테스트 (41건)
docker compose run --rm django python manage.py load_problems
curl -fsS http://localhost/health/                     # {"status":"ok"}
```
합격: 전 서비스 running, `migrate` 무결, 전체 테스트 `OK`, 홈 접속.

| 재인덱싱 | 같은 노트 재임베딩 | 기존 청크 정리, 중복 없음 |

## A.7 배포·운영 검증 (앱/워커 분리 + RDS)

- 필수 운영 설정: `DJANGO_DEBUG=False`, 운영용 `DJANGO_SECRET_KEY`·`INTERNAL_API_KEY`·`OPENAI_API_KEY`, `DJANGO_ALLOWED_HOSTS`(실 도메인/EIP), 앱·워커 동일 RDS `POSTGRES_HOST`.
- 배포는 `deploy/deploy_main.sh`(앱, `compose/main.yml`) / `deploy/deploy_worker.sh`(워커, `compose/worker.yml`) — `git fetch`→`reset --hard origin/main`→`.env` 점검→`compose up --build -d`→헬스 검증.
- 합격: 앱 서버에 postgres·worker 컨테이너 없음, 워커 서버엔 worker만, 앱이 만든 잡을 별도 워커가 RDS에서 선점·처리, 외부 접속은 Nginx만.
- 재기동·재현성: `restart` 후 영속 볼륨(pgdata·chromadata) 유지·정상화, GitHub clone→`.env`→compose로 동일 구조 재현.

> 참고: 초기 개발 CI/CD(`.github/workflows/deploy.yml`, GitHub Actions)는 이력으로 보존(수동 전용). 현재 배포 경로는 위 스크립트로 일원화됨.
