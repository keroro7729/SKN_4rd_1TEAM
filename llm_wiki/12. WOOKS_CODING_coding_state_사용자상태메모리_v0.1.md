# 12. WOOK'S CODING — coding_state (사용자 상태 메모리)

> 실제 구현(`django/codingstate/`, `fastapi_app/services/coding_state.py`) 기준. 발표/설명용 요약 + 대본 포함.

---

## 0. 한 줄 정의

**coding_state = AI가 사용자별 학습 상태를 요약해 계속 갱신하는 "학습 메모리".**
힌트·튜터·오답분석이 매번 이 메모리를 참고해 **개인화**한다. 사용자에게 직접 노출하지 않는 AI 내부값.

> 대부분의 AI 코딩 도구는 "질문 1 → 답변 1"로 끝나 이전 학습을 기억하지 못한다. coding_state는 **기억을 쌓아 다음 학습에 다시 쓰는** 것이 차별점.

---

## 1. 무엇을 담는가 (`CodingState` 모델, 사용자당 1행)

| 필드 | 의미 |
|---|---|
| `summary` | 2~4문장 진단 요약 |
| `thinking_profile` | **사고 추적 메모리** — 어떻게 사고/디버깅하는지(가설 방식·막히는 지점·개념 공백) |
| `level` | 추정 수준 (입문/초급/중급/고급) |
| `strengths` / `weaknesses` | 강한/약한 알고리즘·유형 |
| `recurring_mistakes` | 반복 실수 패턴 |
| `recommended_focus` | 다음에 집중할 학습 주제 |
| `stats_snapshot` | 이번 갱신에 쓰인 집계 원본(감사/diff용) |
| `source_submission_count` / `refresh_count` | staleness 기준 / 누적 갱신 횟수 |

---

## 2. 어떻게 만들어지는가

```
[제출 기록·오답노트·태그·반복실수]  +  [최근 코드·튜터 질문·오답 회고 원문]  +  [직전 메모리]
            └── 결정적 집계(gather_stats) ──┘         └── 사고 추적 입력 ──┘      └ rolling ┘
                                    │
                       FastAPI /ai/coding-state/summarize (gpt-4o-mini, JSON)
                                    │
                         CodingState 저장(요약 + 사고 프로필 + 레벨/강약점 …)
```

- **결정적 집계(Django `gather_stats`)**: 제출·정답률·난이도별·태그 강약·오류 패턴·오답노트 수.
- **사고 추적 입력**: 최근 제출 **코드 스니펫**, 미니튜터에 던진 **질문**, 오답노트 **회고 원문+AI cause**. → "무엇을 아는가"를 넘어 **"어떻게 사고하는가"** 추론.
- **요약 메모리 연속성(rolling)**: 직전 `summary`/`thinking_profile`을 함께 넣어 **처음부터 새로 쓰지 않고 변화점만 갱신**. 지속되는 특성은 표현을 일관되게 유지.
- **프롬프트 규율**: 근거를 짧게 인용, 직전 대비 변화 우선, 데이터가 적으면 과장 금지.

---

## 3. 어떻게 쓰이는가 (소비)

`get_prompt_context(user)`가 상태를 컴팩트 텍스트로 렌더해 **AI 프롬프트의 참고 컨텍스트**로 주입:

- **힌트** — 수준·약점에 맞춘 단계 힌트.
- **미니튜터** — 상태 + 최근 오답 + RAG로 개인화 답변.
- **오답 분석** — 학습자 수준·약점을 반영한 진단.

→ 각 AI가 "이 사용자는 중급, DP 약함, 경계값 실수 반복, 가설을 급하게 세움"을 알고 응답한다.

---

## 4. 갱신 트리거

- **실시간 훅** `ensure_fresh(user)` — 오답노트 완료 시, 신규 제출 5건 게이트(비용 절감).
- **배치** `batch_refresh(...)` — 신규 활동 쌓인 사용자만 골라 갱신(스케줄/오프라인).
  `python manage.py refresh_coding_state --stale [--min-new N] [--limit N]`
- ⚠️ 사용자당 FastAPI(LLM) 동기 호출 → 요청 스레드 금지, 크론/관리 명령 전용.

---

## 5. 다섯 개 전문 Agent 속 위치

하나의 거대한 AI 대신 역할을 **5개 Agent**로 분리, coding_state가 이들을 잇는 공유 메모리:

`테스트케이스 생성` · **`coding_state(학습상태)`** · `힌트` · `오답노트+RAG` · `미니튜터`

(전체 구조: `llm_wiki/11. ...AI_Agent_구조_및_핵심기술_v0.1.md`)

## 6. 발표 대본 (약 60초)

> 기존 AI 코딩 서비스는 대부분 **질문 하나에 답변 하나**로 끝나고, 내가 방금 뭘 틀렸는지는 기억하지 못합니다.
>
> 저희는 제출 기록과 오답노트, 알고리즘 태그, 반복 실수를 계속 모아, AI가 요약한 **'코딩 스테이트'**라는 학습 상태로 만듭니다. 현재 수준, 강점과 약점, 반복 실수, 추천 학습 방향, 그리고 이 학생이 **어떻게 사고하는지**까지 담기죠.
>
> 이 상태는 힌트를 만들 때도, 오답을 분석할 때도 다시 참고됩니다. **AI가 이전 학습을 기억해 다음 학습에 활용하는** 구조입니다.
>
> 이를 위해 하나의 거대한 AI 대신, 테스트케이스 생성·학습상태 분석·힌트·오답분석·대화형 튜터, **다섯 개의 전문 Agent**로 역할을 나눴습니다.

**(선택) 시간 여유 시 한 줄 추가** — 메모리 연속성 강조:
> 게다가 이 상태는 매번 새로 쓰는 게 아니라, 직전 기록 위에 **변화만 얹어 갱신**하는 진짜 '메모리'입니다.
