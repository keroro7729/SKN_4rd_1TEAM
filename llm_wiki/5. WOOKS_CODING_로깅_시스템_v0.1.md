# WOOK'S CODING — 로깅 시스템 가이드 (v0.1)

작성일: 2026-07-03 · 대상: django · fastapi_app · worker

> 목적: **개발·디버깅·AI연구** 로그를 리포 루트 `logs/` 한 곳에 모으고,
> 컨테이너 수명과 무관하게 호스트에서 바로 열람/분석할 수 있게 한다.

---

## 1. 설계 원칙
1. **단일 수집 지점**: 모든 컴포넌트가 리포 루트 `logs/` 아래에 쓴다.
2. **bind mount(≠ named volume)**: compose 가 `./logs:/app/logs` 로 마운트 →
   컨테이너를 지워도 로그는 호스트 `logs/` 에 남아 `jq`/에디터로 즉시 분석 가능.
3. **회전(rotate)**: 각 앱 로그는 `RotatingFileHandler` 5MB × 5개로 무한 증가 방지.
4. **AI 연구용 분리 채널**: LLM/RAG 이벤트는 사람이 읽는 텍스트 로그와 별개로
   `logs/ai/research.jsonl` 에 **구조화(JSON Lines)** 로 축적 → 후속 분석/튜닝에 직결.
5. **로컬/Docker 동일 코드**: 경로는 환경변수 `LOG_DIR` 로 통일(빈 값이면 `<repo>/logs`).

---

## 2. 디렉토리 레이아웃
```
logs/
├─ .gitignore           # 로그 파일 제외, 디렉토리/README 만 커밋
├─ README.md
├─ django/app.log
├─ fastapi/app.log
├─ worker/app.log
└─ ai/research.jsonl    # AI 연구용 (JSON Lines)
```

---

## 3. 환경변수
| 변수 | 기본값 | 설명 |
|---|---|---|
| `LOG_LEVEL` | `INFO` | 루트 로그 레벨 |
| `LOG_DIR` | (빈 값) | 비우면 각 컴포넌트가 `<repo루트>/logs`. **Docker 는 compose 가 `/app/logs` 주입** |

> ⚠️ `.env` 빈 값 gotcha 회피를 위해 코드는 `os.environ.get("LOG_DIR") or (ROOT/"logs")`
> 패턴을 쓴다(빈 문자열이면 폴백이 동작하도록).

### 경로 통일 방식
- 로컬: Django `BASE_DIR.parent/logs`, FastAPI/Worker `parents[1]/logs` = 모두 `<repo>/logs`.
- Docker: WORKDIR `/app` 이라 코드 기준 루트가 `/` 가 되므로, compose 에서
  `LOG_DIR=/app/logs` 를 **명시 주입**하고 `./logs:/app/logs` 를 마운트해 로컬과 결과를 일치시킨다.

## 4. 컴포넌트별 구현

### 4.1 Django (`config/settings.py`)
- `LOG_DIR`, `LOG_LEVEL` 해석 후 표준 `LOGGING` dict 구성(console + RotatingFileHandler).
- 출력: `logs/django/app.log`. `django` 로거와 root 를 파일+콘솔로 연결.

### 4.2 FastAPI (`fastapi_app/logging_setup.py`)
- `setup_logging()`: root 로거에 console + `logs/fastapi/app.log` 부착(중복 호출 안전).
- `log_ai_event(event, **fields)`: `logs/ai/research.jsonl` 에 JSON 한 줄 기록.
  - 전용 로거 `ai_research`, `propagate=False` 로 일반 로그와 분리.
  - `main.py` 에서 `setup_logging()` 호출, 라우터에서 이벤트 기록.
- 현재 계측 지점(스캐폴딩): `/ai/hint`(event=`hint`), `/ai/wrong-note/search`(event=`rag_search`),
  `/analyze`(`analyze`), `/ask`(`note_ask`). STEP-06 에서 프롬프트/응답/지연시간/스코어까지 확장.

### 4.3 Worker (`worker/logging_setup.py`)
- `setup_logging()` → `logging.Logger`("worker"), console + `logs/worker/app.log`.
- `main.py` 의 `print(...)` 를 로거 호출로 대체.

## 5. AI 연구용 로그(research.jsonl) 스키마 컨벤션
공통 필드: `ts`(UTC ISO8601), `event`. 이벤트별 추가 필드는 아래 권장.

| event | 필드(현재/예정) |
|---|---|
| `hint` | problem_id, hint_level, model, (STEP-06) latency_ms, prompt_tokens, completion_tokens |
| `rag_search` | user_id, top_k, hits, (STEP-06) threshold, scores[] |
| `analyze` | (STEP-06) user_id, evidence_count, sections |
| `note_ask` | user_id, hits, (STEP-06) evidence_note_ids[] |

> 원칙: **사람이 읽는 로그(app.log)와 기계가 읽는 로그(research.jsonl)를 섞지 않는다.**
> 분석 예: `jq -c 'select(.event=="hint")' logs/ai/research.jsonl`
